## 15. Acceptance Criteria

The system is accepted when all of the following are true.

Every criterion carries exactly one tier marker:

- **`[crit]`** — **Critical**: a defect means data corruption, financial impact, authentication/authorization failure, data integrity violation, or misleading state that causes wrong user decisions. Verified by automated test. The method matches the contract's surface — unit, integration, E2E or a mix thereof.
- **`[vis]`** — **Visual / Design**: specifies expected behavior that does not guard a critical path. Verified by structural E2E as well as regular human review.
- **`[infra]`** — **Infrastructure / Structural**: a constraint on the deployed environment, the build pipeline, or the source-tree organization. Verified by deployment procedures, CI gates, lint, or code review — not by tests against the running system.

### 15.1 Core

- **AC-1** `[vis]`: The full stack (frontend, backend, database) starts locally with a documented command or minimal command sequence.
- **AC-2** `[vis]`: Kanban view renders 9 columns with all projects in their correct states.
- **AC-3** `[vis]`: Calendar view renders projects with planned dates as colored bars on a month grid.
- **AC-4** `[vis]`: Clicking a project in either view opens the detail panel with all available fields.
- **AC-5** `[crit]`: The [→] button transitions a project to the next state, with a German confirmation dialog. The card moves to the correct column. The change persists across page reloads.
- **AC-6** `[crit]`: Backward transition via the detail panel moves a project to the previous state. The change persists across page reloads.
- **AC-7** `[crit]`: Changing a date in the detail panel updates `plannedStart`/`plannedEnd` and is reflected in both views. The change persists across page reloads.
- **AC-8** `[vis]`: Summary area shows counts for action states and aged buffer items.
- **AC-9** `[vis]`: Clicking a summary indicator from any view switches to the Kanban view and applies the filter to affected projects. Clicking the same active indicator clears the filter without navigating.
- **AC-10** `[vis]`: "X Projekte ohne Termin" counter appears below the calendar.

### 15.2 Visual

- **AC-11** `[vis]`: Action columns are visually distinct from buffer columns.
- **AC-12** `[vis]`: Each state has a consistent color across Kanban dots, calendar bars, and detail badge.
- **AC-13** `[vis]`: Aged buffer items show a `"seit X Tagen"` indicator.
- **AC-14** `[vis]`: Cards display project number, title, customer, dates (or "Kein Termin").
- **AC-15** `[vis]`: Every card shows its `statusChangedAt` date. The date turns bold when the configured aging threshold is exceeded.

### 15.3 Behavioral

- **AC-16** `[crit]`: State transitions only allow forward +1 or backward -1. No skipping.
- **AC-17** `[crit]`: `Erledigt` is terminal — both transition buttons are hidden.
- **AC-18** `[crit]`: `Anfrage` hides the backward transition button.
- **AC-19** `[vis]`: Display dates use German format (DD.MM.YYYY). Date input controls respect the user's browser locale. Calendar week starts Monday.
- **AC-20** `[vis]`: Kanban, Calendar, and the Project Detail Panel render without errors when `customer`, `plannedStart`, `plannedEnd`, `notes`, `assignedWorkers`, or `estimatedValue` are absent.
- **AC-53** `[crit]`: A failed mutation displays a German error message and reverts the optimistic UI update.
- **AC-122** `[vis]`: Modals close on Escape (except while a mutation is in flight, see [AC-131](#153-behavioral)). Form modals submit the primary action on Enter when focus is within the form. Read-only modals accept Escape but do not submit on Enter.
- **AC-123** `[vis]`: Form modals and confirmation dialogs do not close on backdrop click — only via Escape or the explicit cancel action. Non-editing side panels close on backdrop click.
- **AC-131** `[crit]`: While a create, edit, or state-changing mutation is in flight, the form's submit action is disabled, every input in the form is disabled, and the enclosing modal cannot be closed (Escape, close button, backdrop). The lock covers any user-confirmation dialog that precedes request dispatch, so a submit initiated behind an open confirm does not fire a second time. The lock releases when the request resolves (success or failure). Rationale: silent double-dispatch or a close-during-flight produces state the user did not intend — a misleading-state class defect.

### 15.4 Authentication

- **AC-22** `[crit]`: Entering valid credentials and clicking "Anmelden" logs the user in and shows the Kanban view.
- **AC-23** `[crit]`: Entering invalid credentials shows a generic error message in German.
- **AC-24** `[vis]`: The user's display name is shown in the header.
- **AC-25** `[crit]`: Clicking "Abmelden" logs the user out and returns to the login screen.
- **AC-26** `[crit]`: After logout, pressing the browser back button does not reveal project data.
- **AC-27** `[crit]`: A session that expires while the app is open redirects to login with an expiry message.
- **AC-28** `[crit]`: A request with a valid session token for a deactivated user is rejected with an authentication error.
- **AC-52** `[crit]`: An authenticated user can change their own password. A change attempt with an incorrect current password is rejected.

### 15.5 Multi-User

- **AC-29** `[crit]`: Two users logged in simultaneously see each other's changes after refreshing.

### 15.6 Deployment

- **AC-30** `[infra]`: The application is reachable by authorized clients over HTTPS. HTTPS is required by default; the only documented exception is the guarded evaluation mode (see AC-45). Network reachability is scoped by [ADR-0008](../adr/0008-vpn-first-network-access.md); see also AC-49.
- **AC-31** `[infra]`: A CI-built image can be promoted to the hosted environment via manual, pull-based deploy over VPN (see [ADR-0012](../adr/0012-manual-pull-based-deploy-over-wireguard.md)).
- **AC-45** `[crit]`: HTTPS is enforced by default. HTTP-only mode requires an explicit opt-in flag AND a non-production environment (neither alone is sufficient). When HTTP mode is active, the UI shows a non-dismissible warning banner on every page. Enabling the insecure flag in production causes the server to refuse to start. See [ADR-0013](../adr/0013-http-only-evaluation-mode.md).
- **AC-46** `[infra]`: A failed deployment leaves the previously running version running.
- **AC-47** `[infra]`: A previously deployed commit can be redeployed (rollback) by the operator over VPN. See [ADR-0012](../adr/0012-manual-pull-based-deploy-over-wireguard.md).
- **AC-48** `[infra]`: After every deploy, an automated smoke test verifies the health endpoint. Failure aborts the deploy.
- **AC-49** `[infra]`: The hosted environment is reachable only via VPN. See [ADR-0008](../adr/0008-vpn-first-network-access.md).
- **AC-50** `[infra]`: Database and object storage data persist across container restarts and redeploys.
- **AC-51** `[infra]`: Deployments use a specific commit SHA, not a moving tag.

### 15.7 Engineering

- **AC-32** `[infra]`: Code structure and dependency direction follow [architecture.md §11.2](architecture.md#112-responsibility-boundaries). No reverse imports.
- **AC-33** `[infra]`: Mutation boundary per [architecture.md §11.1](architecture.md#111-mandatory-constraints) is enforced. UI components dispatch through the state layer, not directly to the API client.
- **AC-34** `[infra]`: State configuration (labels, colors, thresholds) is centralized, not scattered.
- **AC-35** `[infra]`: Dependency direction is enforced by linter import restriction zones.
- **AC-36** `[infra]`: Linting and formatting pass.
- **AC-37** `[infra]`: Tests defined in section 16 pass. Push/PR gate runs unit, component, and API integration tests. E2E tests run on-demand (see [architecture.md §11.7](architecture.md#117-continuous-delivery-pipeline)).
- **AC-247** `[crit]`: Transport-layer rejections from the API server preserve the 4xx HTTP statusCode native to the failure class and ship a stable machine-readable code, per [api.md §14.4.2](api.md#1442-error-principles): a malformed body declared as `Content-Type: application/json` returns `400 VALIDATION_ERROR`; a payload exceeding the configured maximum returns `413 VALIDATION_ERROR`; a request whose `Content-Type` falls outside the accepted set returns `415 VALIDATION_ERROR`; a request to a URL with no registered handler returns `404 ROUTE_NOT_FOUND`. Pre-existing branches are unchanged: schema-validation rejections remain `422 VALIDATION_ERROR` (with the field-level `details` array preserved), rate-limit rejections remain `429 RATE_LIMITED`, and any error whose statusCode is in the 5xx range — or carries no statusCode — maps to `500 SERVER_ERROR`. Transport-layer 4xx rejections are logged below the operational `error` level. Tiered `[crit]` because the contract guards operator triage: a transport-layer 4xx surfacing as 5xx (response or log) misroutes oncall attention away from genuine server failures.

- **AC-39** `[crit]`: Authentication parameters (session duration) are driven by configuration.

### 15.9 Infrastructure

- **AC-40** `[crit]`: Object storage module successfully uploads and retrieves a file in the deployed environment (see [architecture.md §11.4](architecture.md#114-object-storage-module)).
- **AC-132** `[crit]`: The server runs a periodic job that deletes expired sessions (sessions past their `expiresAt`, see [data-model.md §5.4](data-model.md#54-session)). Rationale: unbounded growth of expired session rows degrades auth-check performance and is an operational data-integrity concern.

### 15.11 Customer Management

- **AC-54** `[crit]`: Creating a customer with a name returns a customer object with a generated ID. Phone, email, address, and notes are optional.
- **AC-55** `[crit]`: Updating a customer changes only the specified fields (PATCH semantics). Passing `null` clears an optional field.
- **AC-56** `[crit]`: Listing customers returns all customers with pagination support. The `search` parameter filters by name (case-insensitive substring match).
- **AC-57** `[crit]`: Getting a customer returns the full customer object and a count of associated projects.
- **AC-58** `[crit]`: A project references a customer via `customerId`. The API returns the full customer object nested in project responses.
- **AC-91** `[crit]`: Deleting a customer with no active projects AND no issued or cancelled invoices on any of its archived projects succeeds (hard delete). Any archived projects are purged atomically with the customer; any draft invoices on those archived projects cascade-delete with the project. The customer no longer appears in list or get responses. Customers whose archived projects carry issued or cancelled invoices are rejected separately under [AC-307](#1511-customer-management).
- **AC-92** `[crit]`: Deleting a customer that has active (non-archived) projects is rejected as a conflict (per the conflict error category in [api.md §14.4.1](api.md#1441-error-categories)). The customer and its projects remain unchanged.
- **AC-93** `[crit]`: Deleting a customer requires `customer:delete` permission. Users without this permission receive an authorization error.
- **AC-154** `[crit]`: The customer-delete confirmation dialog displays a German warning that names the archived-project count when `archivedProjectCount > 0` (sourced from `GET /api/customers/:id`). When the count is 0, the standard confirmation text is used. Rationale: the warning prevents silent destructive data loss — deleting a customer atomically purges those archived projects (and any draft invoices on them) when the customer is deletable per the cascade rule (see [AC-91](#1511-customer-management) and [ADR-0017](../adr/0017-soft-delete-as-board-archive.md)). Customers whose project graph carries any issued or cancelled invoice are blocked separately by [AC-307](#1511-customer-management); the affordance is disabled before this dialog is reached.
- **AC-124** `[crit]`: Creating a customer with a client-supplied `id` persists the row under that id. Replaying the same request (same id, same user-supplied fields) returns the existing row with no duplicate persisted. A concurrent replay of the same id and body results in one insert and one replay, never two rows. See [api.md §14.2.5](api.md#1425-customer-management) for the field-comparison rule.
- **AC-306** `[crit]`: The `Customer` entity carries an optional `ustId` field (USt-IdNr.) ([data-model.md §5.6](data-model.md#56-customer-entity)). `POST /api/customers` accepts the field at create; `PATCH /api/customers/:id` accepts the field with the standard `null`-clears semantics; `GET /api/customers/:id` returns the field when set. The field participates in the customer idempotent-replay field comparison ([api.md §14.2.5](api.md#1425-customer-management)). The field is **not** required by the customer form regardless of mode; the requiredness gate runs at invoice issuance against `recipient.ustId` only when the draft's `taxMode = 'reverse_charge'` (per AC-289 extension). Pre-fills `Invoice.recipient.ustId` on draft creation when set.
- **AC-307** `[crit]`: `DELETE /api/customers/:id` is rejected with `409 CONFLICT` and `code = CUSTOMER_HAS_INVOICES` when any of the customer's projects (active or archived) carries at least one issued or cancelled `Invoice` row. Draft invoices do not trigger the rejection — they cascade-delete with their project on the customer's atomic-purge cascade. `details.invoiceCount` carries the count of issued + cancelled rows. The customer row, projects, and invoices are unchanged on rejection. `GET /api/customers/:id` exposes the count as `invoiceCount` (issued + cancelled only) so the destructive-warning UI ([ui/management.md §8.9.5](ui/management.md#895-delete-customer)) renders the matching disabled state. A customer whose projects (active + archived) carry zero issued or cancelled invoices behaves as before (existing AC-91 / AC-92); a customer whose archived projects carry only draft invoices is deletable, and the cascade removes those drafts alongside the projects.

### 15.12 Project Management

- **AC-59** `[crit]`: Creating a single project with `number`, `title`, and `customerId` returns a project in the first workflow state. Optional fields default appropriately.
- **AC-60** `[crit]`: Updating a project changes the specified fields. Status and project number are not changeable via update (status uses transitions; number is immutable).
- **AC-61** `[crit]`: Soft-deleting a project marks it as deleted. The project no longer appears in list results, views, or exports.
- **AC-62** `[crit]`: Project number is unique. Creating a project whose `number` collides with an existing project is rejected with status 409 and error code `CONFLICT`. The German error message names the conflicting number. Distinct from the `IDEMPOTENCY_CONFLICT` path in [AC-127](#1517-data-integrity).
- **AC-125** `[crit]`: Creating a project with a client-supplied `id` persists the row under that id. Replaying the same request (same id, same body) returns the existing row with no duplicate persisted. A concurrent replay of the same id and body results in one insert and one replay, never two rows. See [api.md §14.2.2](api.md#1422-projects) for the field-comparison rule.
- **AC-151** `[crit]`: `GET /api/projects` accepts an `includeArchived` query parameter (boolean, default false). When false or omitted, the result excludes soft-deleted (`deleted = true`) rows. When true, the result includes them with their archive state preserved. The parameter AND-composes with the other filters (see [api.md §14.2.2](api.md#1422-projects)). Rationale: the default-safe boundary prevents leaking archived rows into contexts that expect active-only (misleading state).
- **AC-155** `[crit]`: `DELETE /api/projects/:id/purge` hard-deletes a project that is already archived (`deleted = true`). Assigned-worker rows (`project_workers`) are removed via cascade. The row is absent from subsequent list and get responses (including `includeArchived=true`). See [ADR-0017](../adr/0017-soft-delete-as-board-archive.md).
- **AC-156** `[crit]`: `DELETE /api/projects/:id/purge` on a non-archived project is rejected with 409 Conflict (per the conflict error category in [api.md §14.4.1](api.md#1441-error-categories)). The project is unchanged. The German error message directs the user to archive the project first.
- **AC-157** `[crit]`: `DELETE /api/projects/:id/purge` requires the `project:purge` permission. Callers without it receive 403 `NOT_PERMITTED`. The existing `project:delete` permission is not sufficient — purge is a narrower, strictly more destructive operation.
- **AC-158** `[crit]`: `DELETE /api/projects/:id/purge` on a non-existent project ID returns 404 (standard not-found contract).
- **AC-278** `[crit]`: `POST /api/projects` accepts an optional `siteAddress` object with `street`, `zip`, `city` (see [data-model.md §5.1](data-model.md#51-project-entity)). When supplied as a fully populated object, the value is persisted on the row. When omitted or supplied as `null`, the row's `siteAddress` is `null` and the project's site is the customer's billing address by the fallback rule in [data-model.md §5.1](data-model.md#51-project-entity). The created row's `siteAddress` round-trips on `GET /api/projects/:id`. `siteAddress` participates in the field comparison for idempotent replay ([api.md §14.2.2](api.md#1422-projects) design notes); a replay with the same `id` carrying a `siteAddress` whose `(street, zip, city)` triple matches component-wise is treated as a successful replay, while a divergent triple or a divergence between null-and-set is rejected with `IDEMPOTENCY_CONFLICT` ([AC-127](#1517-data-integrity)).
- **AC-279** `[crit]`: `PATCH /api/projects/:id` accepts `siteAddress`. Omitting the field leaves the stored value unchanged (PATCH semantics). Passing `siteAddress: null` clears the stored value — the project's site reverts to the customer's billing address per the fallback rule. Passing a fully populated `{ street, zip, city }` overwrites the stored value. The clearing semantics mirror the customer-address PATCH-null rule pinned by [AC-55](#1511-customer-management). The mutation produces an `audit_log` row through the single-write-path helper per [AC-177](#1523-audit-log).
- **AC-308** `[crit]`: `DELETE /api/projects/:id/purge` is rejected with `409 CONFLICT` and `code = PROJECT_HAS_INVOICES` when the project carries at least one issued or cancelled `Invoice` row (immutable per [§6.14](data-model.md#614-immutability-of-issued-invoices); §147 AO retention). Draft invoices on the project do **not** trigger the rejection — they cascade-delete with the project alongside `project_workers` rows. `details.invoiceCount` carries the count of issued + cancelled rows. The project row, its archive state, and all invoices are unchanged on rejection. Distinct from AC-156 (`CONFLICT` for non-archived) so the client renders the targeted retention-notice message rather than the archive-first hint. The error code is **not** collapsed into the existing `CONFLICT` to keep the client's error-routing surgical. Purging an archived project that carries only draft invoices succeeds and removes those drafts via FK cascade.

### 15.13 User Management

- **AC-63** `[crit]`: An owner can create a new user account with username, display name, password, and roles. The created user can log in with the provided credentials.
- **AC-64** `[crit]`: An owner can update a user's display name, roles, and email. Username is immutable — attempts to change it are rejected.
- **AC-65** `[crit]`: An owner can deactivate a user. Deactivation invalidates all sessions for that user. The deactivated user cannot log in.
- **AC-66** `[crit]`: An owner can reactivate a previously deactivated user. The reactivated user can log in again.
- **AC-67** `[crit]`: An owner can reset another user's password. The new password must meet the password policy. The operation invalidates all sessions for the target user.
- **AC-68** `[crit]`: A user cannot deactivate themselves. The API rejects self-deactivation with an error.
- **AC-69** `[crit]`: Only users with `user:manage` permission (owner role) can create, update, deactivate, reactivate, or reset passwords. Other roles receive an authorization error.

### 15.14 Data Exchange

- **AC-133** `[crit]`: The unified export rejects unauthenticated requests and requests from users without `data:export` with an authorization error.
- **AC-134** `[crit]`: The unified import rejects unauthenticated requests and requests from users without `data:restore` with an authorization error.
- **AC-135** `[crit]`: The export envelope contains `schema_version`, `exported_at`, and all customers, projects, and project-worker assignments, including archived (soft-deleted) rows with archive state preserved.
- **AC-136** `[crit]`: An import whose envelope `schema_version` does not equal the current one is rejected with a specific error code and performs no writes.
- **AC-137** `[crit]`: An import into an empty database succeeds, preserves IDs exactly, and runs as a single transaction (all-or-nothing).
- **AC-138** `[crit]`: An import into a non-empty database is rejected unless the override flag is set in the request.
- **AC-139** `[crit]`: An import with the override flag into a non-empty database wipes existing business data and restores atomically. If any row fails validation, no state change is persisted.
- **AC-140** `[crit]`: A dry-run import performs full validation, returns a preview, and makes no writes. Subsequent reads show no state change.
- **AC-141** `[crit]`: A full roundtrip — seed → export → wipe → import → export — produces byte-identical export output **for the text-row slice** (customers, projects, project-worker assignments, including archived rows; envelope `attachments[]` is unaffected by the import path under [AC-253](#1514-data-exchange), so re-export drops attachments on the seeded dataset by construction). Binary roundtrip rides the takeout-zip path covered by [AC-259](#1514-data-exchange) — out of scope for AC-141. Verified in CI on every build.
- **AC-142** `[vis]`: The Daten navigation tab is hidden for users without `data:export`. Specialization of [AC-121](#1516-management-views) for this tab.
- **AC-160** `[crit]`: `POST /api/import?override=true` into a non-empty database requires a `confirmation_phrase` field in the request body matching the configured phrase **[C]**. A missing or non-matching phrase is rejected with error code `RESTORE_CONFIRMATION_MISMATCH` and performs no writes. Dry-run requests and override-into-empty-target requests are exempt. Comparison is case-sensitive after trimming leading/trailing whitespace.
- **AC-161** `[vis]`: The restore UI renders a destructive-action confirmation input only when the dry-run preview indicates the target database is non-empty. The commit action stays disabled until the typed value matches the configured phrase **[C]**. Client-side disabling is a UX affordance; the server remains authoritative (see [AC-160](#1514-data-exchange)).
- **AC-162a** `[crit]`: On the commit path, `POST /api/import` rejects an envelope whose user-id references (`customers.createdBy`, `customers.updatedBy`, `projects.createdBy`, `projects.updatedBy`, `project_workers.userId`) point to users absent from the target database. The response is HTTP 422 with error code `MISSING_USER_REFS`. `details.missingUserIds` is the deduplicated list of absent user ids; `details.references` carries one `{ path, userId }` entry per offending envelope reference site (duplicates across sites produce duplicate entries). Null or missing audit-field values do not trigger the code. No writes are persisted; atomicity with the wipe phase when `override=true` is already covered by [AC-139](#1514-data-exchange) (and [AC-137](#1514-data-exchange) for the single-transaction guarantee). See [api.md §14.2.4](api.md#1424-unified-data-exchange) and [api.md §14.4.1](api.md#1441-error-categories).
- **AC-162b** `[crit]`: On the dry-run path (`dry_run=true`), missing-user-reference issues and intra-envelope issues are both evaluated regardless of intra-envelope state and surfaced together in the preview. The gate that restricts the missing-user check to intra-consistent envelopes applies only to the commit path. No writes occur on dry-run. See [api.md §14.2.4](api.md#1424-unified-data-exchange).
- **AC-162c** `[crit]`: On the commit path, intra-envelope referential integrity is reported first under `VALIDATION_ERROR`. The missing-user-reference check runs only when the envelope is intra-consistent, and `MISSING_USER_REFS` is raised only on an intra-consistent envelope. A single commit-path response never carries both codes. See [api.md §14.2.4](api.md#1424-unified-data-exchange).
- **AC-248** `[crit]`: The binary-descriptors surface ([api.md §14.2.4](api.md#1424-unified-data-exchange)) returns `{ entries: BinaryDescriptor[], nextCursor: string | null, totalCount: number, totalSizeBytes: number }` for every `status='ready'` attachment row. `pending` and `hidden` rows are excluded by construction. Each `entries[i]` carries `attachmentId`, `projectId`, `projectNumber`, `projectTitle`, `fileName`, `sizeBytes` always; AND either the fetchable triple `originalUrl` (presigned-GET on the ciphertext object) + `originalDekMaterial` (base64 of the 32-byte DEK that decrypts `originalUrl`) + `expiresAt` (ISO 8601 at the configured presigned-URL TTL **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings))) OR the error tag `error = 'DEK_UNWRAP_FAILED'` — never both, never neither. Thumbnails are not included. `totalCount` is the row count and `totalSizeBytes` the summed plaintext `sizeBytes` of those rows, both server-computed at first-page composition and returned identically on every subsequent page within the iteration. Ordering is ascending `(createdAt, id)`: a row with `(createdAt, id)` strictly greater than the cursor at page-composition time is surfaced in a later page; a row whose status changed away from `ready` after the cursor passed it is not retroactively included or excluded. Pagination defaults and ceiling come from the export-all per-page descriptor limit **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings)). The endpoint writes no `audit_log` row, parity with the existing `Export` operation in the same section ([AC-133](#1514-data-exchange)) — the read action has no domain-state change, and the `audit_log` invariant ([AC-177](#1523-audit-log)) ties audit rows to mutations. Permission: `data:export` (owner / office under the default matrix); unauthenticated requests return `401 UNAUTHENTICATED` / `SESSION_EXPIRED`, callers without `data:export` return `403 NOT_PERMITTED`. The descriptor surface is unscoped because the admitting roles are unscoped under `attachmentScopeForCaller` ([AC-217](#1526-attachments)). A malformed `after` cursor or `limit` outside the configured per-page bounds returns `422 VALIDATION_ERROR`. A wholesale envelope-unwrap failure on the page (operator's binary `age` identity not loaded) returns `500 SERVER_ERROR`. The browser-side decrypt + streaming-zip assembly is out of spec scope — the wire contract is what this AC pins.
- **AC-249** `[vis]`: The Export action on the Daten view ([ui/daten.md §8.11.1](ui/daten.md#8111-export)) renders a pre-flight confirmation dialog naming the attachment count from `totalCount` and the aggregate plaintext size from `totalSizeBytes` — both server-computed and surfaced as a single up-front readout (no growing-under-the-user behavior). During the export it surfaces files-done / total, bytes-done / total (computed from the iteration's `entries[*].sizeBytes` against `totalSizeBytes`), the current file name, and an **"Abbrechen"** action; clicking Abbrechen halts the in-flight fetch, tears down the streaming-zip, and closes the dialog immediately. Any partially-written download is the user's to discard. Below the configured mobile-warning breakpoint **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings)) the dialog additionally renders a non-blocking warning `"Für Desktop-Nutzung gedacht; Downloads können sehr groß sein."` — the user can still proceed. The resulting download is named `projekt-manager-export-<YYYY-MM-DD>T<HH-mm-ss>.zip`. An unsupported runtime (no controlling Service Worker, no transferable-stream support) surfaces the generic export-failed message and refuses the action — see [ui/daten.md §8.11.1](ui/daten.md#8111-export). Permission: `data:export` — users without it do not see the action ([AC-150](#1521-role-scoping)).
- **AC-250** `[crit]`: The zip produced by [AC-249](#1514-data-exchange) carries `data.json` (the unified envelope from [AC-135](#1514-data-exchange)) and `manifest.json` (per [AC-252](#1514-data-exchange)) at the root, plus every `status='ready'` attachment whose descriptor in the iteration carried the fetch triple (NOT `error = 'DEK_UNWRAP_FAILED'`, see [AC-248](#1514-data-exchange)) as plaintext bytes under `attachments/<projektnummer>-<projekt-titel>/<attachment-id>-<dateiname>`. Each path component is sanitised by the same rules pinned in [AC-245](#1526-attachments) for filename validation: 255-character ceiling per component, no control characters (`\x00`–`\x1F`, `\x7F`), no path separators (`/`, `\`), no double-quotes (`"`); a violating character is replaced with an underscore (`_`) and a violating length is truncated. `(projektnummer, dateiname)` collisions across rows are defused by the prepended `attachment-id`. The plaintext bytes at each `attachments/...` path equal the bytes the user uploaded for that attachment (recovered by AES-256-GCM-decrypting the ciphertext object with `originalDekMaterial`, [ADR-0024](../adr/0024-binary-attachment-e2e-encryption.md)).
- **AC-251** `[vis]`: A per-file failure inside the [AC-249](#1514-data-exchange) flow does not abort the export. An entry whose descriptor carries `error = 'DEK_UNWRAP_FAILED'` (see [AC-248](#1514-data-exchange)) is logged and skipped without attempting a fetch. An entry whose presigned `expiresAt` has passed before the fetch triggers ONE re-fetch of the affected descriptor page (the client re-issues the same `after` cursor); on a second expiry the entry is logged and skipped. A fetch error (storage 4xx / 5xx, network failure on the ciphertext fetch) likewise logs and skips the entry. The post-export summary surfaces a skipped-row count `"X Dateien übersprungen"` when at least one entry was skipped, regardless of skip cause. Partial completeness is the user-visible contract.
- **AC-252** `[crit]`: The zip produced by [AC-249](#1514-data-exchange) carries `manifest.json` at the root with the shape `{ manifestVersion: 1, exportedAt: ISO 8601 string, totalFiles: number, totalBytes: number, files: ManifestEntry[] }`. Each `files[i]` carries `zipPath` (the entry's path inside the same zip), `sizeBytes` (byte length of the entry's bytes inside the zip), and `sha256` (hex-lowercase SHA-256 of the entry's bytes inside the zip, exactly 64 characters from `[0-9a-f]`). Attachment entries additionally carry `attachmentId` (the source row id); the `data.json` entry does not. The first entry is `data.json`; subsequent entries appear in the same order the attachments landed in the zip. `totalFiles` equals `files.length`; `totalBytes` equals the sum of `files[*].sizeBytes`. The manifest does NOT list itself. The manifest does NOT list attachments skipped per [AC-251](#1514-data-exchange) — it reflects what is IN the zip, not what was attempted. Verification invariant: for every `files[i]`, `sha256` equals SHA-256 of the bytes at `zipPath` inside the same zip. The manifest is computed client-side; the server has no involvement. An external verifier can produce `sha256sum -c` input from the manifest with one projection: `jq -r '.files[] | "\(.sha256)  \(.zipPath)"' < manifest.json`.
- **AC-253** `[crit]`: `POST /api/import` never inserts rows into the `attachments` table on any input. The endpoint accepts only the text-row body (`customers`, `projects`, `project_workers`, `schema_version`, `exported_at`); a request body carrying an `attachments` key is rejected with `422 VALIDATION_ERROR` and no writes occur. The body-shape contract removes the silent-loss bug at the source: pre-fix imports accepted the envelope's `attachments[]` and inserted rows whose wrapped DEKs were unwrappable on the importing instance, so every imported attachment download returned 404. Verified by an integration test that submits a body with an `attachments` key under every combination of `dry_run` and `override` and asserts each call returns `422 VALIDATION_ERROR` with no row changes; a parallel call without the `attachments` key on the same fixture proceeds normally.
- **AC-254** `[crit]`: `POST /api/import?override=true` into a non-empty target truncates the `attachments` table inside the same transaction that wipes existing customer / project / project-worker rows, atomically with the restore. After a successful override-import, the `attachments` table is empty. The truncate is unconditional under `override=true` regardless of envelope `attachments[]` content — the takeout-zip restore mechanics ([AC-256](#1514-data-exchange)) re-upload attachments through `init` + presigned PUT + `complete` after this call returns, not via the import endpoint. Backing B2 objects from the prior state are not touched by the import path; the orphan-reaper infrastructure ([data-model.md §6.11](data-model.md#611-attachment-orphan-reaper)) reaps them on its own cadence. Verified by an integration test that seeds attachment rows, calls `/api/import?override=true` with the configured confirmation phrase, and asserts the `attachments` table is empty post-call.
- **AC-255** `[crit]`: `POST /api/projects/:id/attachments/init` carrying an optional `restore: { id, createdBy, createdAt }` block ([api.md §14.2.11](api.md#14211-attachments)) requires the caller to hold **both** `data:restore` AND `attachment:write`. A caller authenticated and holding `attachment:write` alone (the standard upload path) but missing `data:restore` receives `403 NOT_PERMITTED` and no row is persisted. A caller missing `attachment:write` receives `403 NOT_PERMITTED` per the existing upload-permission contract. An init call without a `restore` block is unaffected by this AC (regression coverage in [AC-258](#1514-data-exchange)). The double-permission gate mirrors the destructive-action policy on `/api/import` ([AC-134](#1514-data-exchange)): `attachment:write` admits the upload at all; `data:restore` admits the override on server-managed identity fields.
- **AC-256** `[crit]`: `POST /api/projects/:id/attachments/init` carrying a valid `restore: { id, createdBy, createdAt }` block persists the supplied values on the resulting `attachment` row instead of the server-generated defaults. Validation runs before any state change: `restore.id` MUST parse as a UUID, `restore.createdAt` MUST parse as ISO 8601, `restore.createdBy` MUST reference an existing `users` row. After validation passes, the `attachment` row carries `id == restore.id`, `createdBy == restore.createdBy`, `createdAt == restore.createdAt`; everything else (the per-blob wrapped envelope, the `attachment:add` audit row, the presigned-PUT descriptors) is identical to the standard init path ([AC-245](#1526-attachments)). The supplied values do NOT bypass any other init-time check (MIME whitelist, label enum, size cap, fileName sanitization, DEK validation) — those run unchanged.
- **AC-257** `[crit]`: `POST /api/projects/:id/attachments/init` with a `restore` block returns `422 VALIDATION_ERROR` and persists no row on each of the bad-input branches: (a) `restore.id` not parseable as a UUID, (b) `restore.createdBy` referencing a user absent from the `users` table, (c) `restore.createdAt` not parseable as ISO 8601. The error code is the standard `VALIDATION_ERROR`; `details` names the offending `restore.<field>`. A `restore.id` that parses but collides with an existing `attachment.id` returns `409 CONFLICT` and persists no row — under the takeout-zip restore mechanics ([AC-254](#1514-data-exchange) leaves the table empty), a collision indicates a client-orchestrator logic error, not a recoverable race.
- **AC-258** `[crit]`: `POST /api/projects/:id/attachments/init` without a `restore` block is unchanged by the import-mode work — regression coverage. The endpoint mints `id` server-side, sets `createdBy` from the session, and stamps `createdAt` from the server clock, exactly as before. The caller is gated on `attachment:write` only ([AC-245](#1526-attachments)); `data:restore` is not required and confers no extra capability on this path. Verified by an integration test that drives the standard upload path (no `restore` block) as a `data:restore`-less office user and asserts the resulting row carries server-generated identity fields.
- **AC-259** `[crit]`: A full takeout roundtrip — seed dataset (with photo and binary attachments) → Export ([AC-249](#1514-data-exchange)) → wipe via `/api/import?override=true` ([AC-254](#1514-data-exchange)) → Import ([AC-260](#1514-data-exchange)) on the now-empty target — restores every attachment row at byte-equal plaintext. For each restored attachment: the row's `id`, `createdBy`, `createdAt` match the source row exactly; the plaintext bytes recovered from the importing instance via `download-url` ([AC-241](#1526-attachments)) byte-equal the bytes the seed uploaded; for photo rows the thumbnail is regenerated by the client image pipeline ([ADR-0024](../adr/0024-binary-attachment-e2e-encryption.md)) on the importing instance and is renderable through `/encrypted-storage/<projectId>/<attachmentId>.thumbnail` ([AC-243](#1526-attachments)). The thumbnail bytes need NOT byte-equal the source thumbnail — thumbs are regenerated, not transferred (the takeout artifact carries plaintext originals only).
- **AC-260** `[crit]`: The Import action ([ui/daten.md §8.11.2](ui/daten.md#8112-import)) verifies the manifest's structural shape and zip-entry coverage parity before any write (step 1). Per-entry SHA-256 verification happens in step 5 (canonical, hashed once on read just before encrypt). A missing entry referenced by `manifest.json`, an extra zip entry not listed in `manifest.json`, or a malformed top-level shape aborts the import in step 1 — no text-leg request is dispatched, no attachment is uploaded. The dialog renders a German error naming the offending `zipPath`. Verified by an integration test on the orchestrator with a fixture zip whose manifest is missing a `zipPath` claimed by a present zip entry; assert the run aborts before dispatching the text-leg request.
- **AC-261** `[crit]`: The Import orchestrator tracks every committed `attachmentId` (each one observed via a successful `complete` response) and rolls back on any fatal mid-stream failure. The fatal set per [ui/daten.md §8.11.2](ui/daten.md#8112-import) step 7: a per-file SHA-256 mismatch detected in step 5 (the canonical zip-integrity check), a user-initiated cancel, and a text-leg rejection arriving after per-attachment work has started. Text rows committed by the text-leg ([AC-254](#1514-data-exchange)) are NOT rolled back — they sit at the same boundary the standard `/api/import` already commits. Verified by an integration test driving a fixture zip whose third attachment carries a manifest SHA-256 that does not match its bytes; assert the first two committed ids each receive a DELETE call and the run aborts on the third entry's hash check before any further upload.
- **AC-262** `[crit]`: The Import orchestrator rejects a takeout zip whose `data.json` `schema_version` is not equal to the importing instance's pinned value. The rejection happens client-side during the file-picker validation step ([ui/daten.md §8.11.2](ui/daten.md#8112-import) step 1) before any text-leg request is dispatched. A defense-in-depth check on the server-side text-leg also rejects a mismatch with error code `SCHEMA_VERSION_MISMATCH` (parity with [AC-136](#1514-data-exchange) on the legacy restore form). ### 15.21 Role Scoping

These criteria pin the per-role read surface described in [index.md §4.2](index.md#42-users) and [api.md §14.3](api.md#143-authorization-rules). They describe behavior, not mechanism — the implementation may realize scoping via a query filter, a repository-layer predicate, or any equivalent approach, as long as the observable contract holds.

- **AC-145** `[crit]`: A worker's `GET /api/projects` returns only projects where the caller is recorded in `project_workers`. A worker assigned to no project receives an empty list. Owner and office receive every non-deleted project (unchanged). Verified by API integration tests using seed users with deterministic assignments.
- **AC-146** `[crit]`: A worker's `GET /api/customers` returns only customers referenced by at least one project where the caller is recorded in `project_workers`. A customer reachable to the worker exclusively through a soft-deleted project is excluded. Owner and office receive every customer (unchanged) — the soft-delete status of a customer's linked projects does not restrict the owner/office customer list, which is the parity clause to AC-145's "every non-deleted project (unchanged)".
- **AC-147** `[crit]`: A worker's `GET /api/projects/:id` succeeds only when the caller is recorded in `project_workers` for that project; otherwise the request is rejected with an authorization error (`NOT_PERMITTED`). The server must not respond with `NOT_FOUND` for a project that exists but the worker is not assigned to — the two outcomes are distinguishable to the caller. This is acceptable under the project's threat model: callers are internal, authenticated users and project IDs are UUIDs that are not enumerable from outside, so resource existence is not a secret at the role boundary (compare with [api.md §14.4.1](api.md#1441-error-categories)'s `NOT_PERMITTED` vs `NOT_FOUND` distinction). Owner and office are unaffected.
- **AC-148** `[crit]`: A worker's `GET /api/customers/:id` succeeds only when the customer is referenced by at least one project where the caller is recorded in `project_workers`; otherwise the request is rejected with an authorization error (`NOT_PERMITTED`). Owner and office are unaffected.
- **AC-149** `[vis]`: A manual URL entry to a path the caller's role is not permitted to access (for example, a worker navigating to `/kunden`) presents an explicit not-permitted error surface in the client: a visible error message indicating the access is denied, no redirect to another view, and the URL in the address bar remains unchanged. This error surface is distinct in presentation from an API 403 returned mid-interaction (see [ui/behavior.md §9.5](ui/behavior.md#95-asynchronous-mutation-behavior)); this criterion covers the route-entry case. Coverage spans every role × every path not permitted for that role under the matrix in [AC-75](#1515-navigation).
- **AC-150** `[vis]`: The Export and Import controls inside the Daten view are rendered only when the caller holds the corresponding permission — Export ([ui/daten.md §8.11.1](ui/daten.md#8111-export)) requires `data:export`; Import ([ui/daten.md §8.11.2](ui/daten.md#8112-import)) requires `data:restore`. This is defense-in-depth against the nav-level gate in [AC-142](#1514-data-exchange): even if the Daten route becomes reachable without the permission (e.g. a bug in the nav gate, a manual URL probe, a client build drift), the component itself must not present the action to a caller who cannot use it. The server remains authoritative ([AC-133](#1514-data-exchange) for export, [AC-134](#1514-data-exchange) for import).

### 15.22 Backup and Recovery

These criteria pin the Layer 2 backup-and-drill cycle ([architecture.md §11.10](architecture.md#1110-full-state-backup-layer-2), [ADR-0020](../adr/0020-layer-2-encrypted-r2-backups-with-operator-loaded-drills.md)). Status fields referenced below are defined in [data-model.md §5.9](data-model.md#59-backup-status-entity).

- **AC-165** `[crit]`: A backup run whose Tier 1 verify-on-create produces a per-table manifest that does not equal the source manifest fails the run. No artifact is uploaded to the off-site object store; `meta_backup_status.lastBackupOk` is `false` and `lastError` identifies the failing table.
- **AC-166** `[crit]`: A backup run whose Tier 1 verify-on-create matches the source manifest uploads the encrypted dump artifact and its encrypted manifest sidecar to the off-site object store; `meta_backup_status.lastBackupOk` is `true` and `lastBackupAt` equals the run timestamp.
- **AC-167** `[crit]`: No backup artifact written to the off-site object store is readable without the operator's private decryption material. Both the dump and the manifest sidecar are encrypted before upload; a run that cannot produce an encrypted artifact fails and uploads nothing. Tool choice lives in [ADR-0020](../adr/0020-layer-2-encrypted-r2-backups-with-operator-loaded-drills.md).
- **AC-168** `[crit]`: When the operator's private identity is absent from tmpfs, the Tier 2 verify-on-cycle drill is skipped. `meta_backup_status.lastDrillAt` is not advanced and `lastDrillOk` is not coerced from its prior value. A skipped drill is not reported as a failure.
- **AC-169** `[crit]`: Every backup run upserts `meta_backup_status` in the application DB AND writes the status mirror object to the off-site object store with the same `lastBackupAt`, `lastBackupOk`, `lastDrillAt`, `lastDrillOk`, `lastError`, and `updatedAt` values. If the mirror-object write fails after the encrypted artifacts have been uploaded, the artifacts remain in place under the bucket's immutability window, the failure is recorded in `meta_backup_status.lastError`, and the artifacts are subject to the normal retention lifecycle.
- **AC-170** `[vis]`: On every authenticated surface the backup-freshness badge is visible only to callers with role `owner`. Other authenticated roles and the unauthenticated login screen do not render the badge. The badge renders as a fixed-size icon at every state; colour (green / amber / red / muted-unknown) is the only signal of severity, so layout does not shift between readings. When the `meta_backup_status` row carries `lastBackupAt`, the tooltip and tap toast append the run timestamp formatted in German (`HH:mm EEE dd.MM.yyyy`) so the operator can tell when the reading was earned. The `unknown` and `backup-never-run` states omit the timestamp — no run to point at.
- **AC-171** `[crit]`: When the status source is unreachable (DB down or the status mirror object not retrievable), the backup-freshness badge renders an explicit "Status unbekannt" state. When `lastDrillAt` is absent (no Tier 2 drill has ever succeeded or failed), the badge renders as drill-stale at the red threshold. The badge is not silently hidden and no stale-but-green state is shown. Rationale: misleading state is a critical defect class per [ADR-0014](../adr/0014-ac-tier-system-critical-vs-design.md).
- **AC-172** `[infra]`: The `backup` compose service is defined in the compose file with the configured interval **[C]** expressed in cron form, and starts with the rest of the stack. Verified by compose-file review — the tier reflects that the observable is the source tree, not a runtime check.
- **AC-173** `[infra]`: The backup bucket has object-lock retention AND a lifecycle rule, both configured per the canonical values in [ADR-0020 §Retention](../adr/0020-layer-2-encrypted-r2-backups-with-operator-loaded-drills.md#retention). Verified by infrastructure review against the runbook; direct automated assertion against the provider is not required. Provider and bucket name live in [ADR-0020](../adr/0020-layer-2-encrypted-r2-backups-with-operator-loaded-drills.md) and the runbook.
- **AC-174** `[crit]`: The per-table content checksum is deterministic across runs on identical data. Two backup runs against the same database state produce byte-equal manifest values for every table. Rationale: non-deterministic checksums invalidate Tier 1 and Tier 2 comparison.
- **AC-175** `[infra]`: The operator's private decryption identity is never persisted to disk on the VPS. The drill-key load path writes the identity only into a tmpfs mount; a reboot removes it. Verified by the runbook procedure and by repository-level review of the drill-key load script. Tool choice and script name live in [ADR-0020](../adr/0020-layer-2-encrypted-r2-backups-with-operator-loaded-drills.md).
- **AC-176** `[crit]`: The owner's backup-freshness badge reads `BackupStatus` from the authenticated `GET /api/auth/me` response (`backupStatus` field, owner-only — see [api.md §14.2.1](api.md#1421-authentication--user-profile)). When the status row is unreachable the field is omitted, which the client maps to the explicit "Status unbekannt" state per AC-171; a silently-empty surface is forbidden. No public `/api/backup/status` endpoint is exposed — the badge has no unauthenticated consumer.

### 15.23 Audit Log

These criteria pin the single-write-path audit surface ([ADR-0021](../adr/0021-audit-log-and-notifications-single-write-path.md), [data-model.md §5.10](data-model.md#510-audit-log-entity), [api.md §14.2.8](api.md#1428-audit-log), [ui/workflow-views.md §8.4.1](ui/workflow-views.md#841-activity-feed), [ui/management.md §8.13](ui/management.md#813-audit-view)).

- **AC-177** `[crit]`: Every mutation on an audited entity (any `AuditEntityType` value — see [data-model.md §5.10](data-model.md#510-audit-log-entity)), **except the non-audited self-preference writes documented in [data-model.md §5.10](data-model.md#510-audit-log-entity) "Non-audited self-preference writes on `UserAccount`"**, produces exactly one `audit_log` row written in the same database transaction as the domain-state change. A domain-state change without its audit row is not possible (save the carve-out); an audit row without its state change is not possible. Row carries a non-null `actorId` AND `actorKind = 'user'` when an authenticated caller drove the mutation; a non-null `actorReason` AND `actorKind = 'system'` and null `actorId` when the write originates from an unattended path. Concurrent mutations on the same entity produce serialized audit rows whose order matches the committed state-change order. Each audit row's `payload.before` reflects the row state read inside the mutation's own transaction before the write.
- **AC-178** `[crit]`: The first-run admin bootstrap ([ADR-0010](../adr/0010-first-run-admin-bootstrap.md)) writes an `audit_log` row with `actorKind = 'system'`, null `actorId`, a non-empty `actorReason` (e.g., `"first-run-bootstrap"`), `entityType = 'user'`, and `action = 'create'`. The database's CHECK constraint rejects a system entry whose `actorReason` is null or empty. Rationale: without a non-empty reason, the bootstrap mutation is invisible in the activity feed, leaving the first administrator's creation unaccounted for.
- **AC-179** `[infra]`: A CI-enforced static architecture check fails the build when an audited-table mutation (insert, update, or delete on any audited table) is authored outside the service-layer's single-write-path helper ([ADR-0021](../adr/0021-audit-log-and-notifications-single-write-path.md)). The check's audited-table set is the `AuditEntityType`-derived set from [data-model.md §5.10](data-model.md#510-audit-log-entity) — `attachment` is a full member of `AuditEntityType`, so the `attachment` table is wired into the check through the same derivation as every other audited table (no carve-out entry, no hand-maintained append). A new `AuditEntityType` value whose corresponding table is not wired into the check fails CI. `audit_log` itself is not in the audited set: `mutate()` is its legitimate writer, and its integrity is guaranteed by the append-only invariant ([§5.10](data-model.md#510-audit-log-entity)) plus the retention-cleanup allowlist. The check allowlists migrations, the unified restore path ([api.md §14.2.4](api.md#1424-unified-data-exchange)), business-data seed loaders, the retention cleanup job ([data-model.md §6.10](data-model.md#610-audit-log-retention)), the attachment complete transition (state-machine finalize, no audit row — [AC-219](#1526-attachments)), and the attachment orphan-reaper ([data-model.md §6.11](data-model.md#611-attachment-orphan-reaper)). Each allowlist addition is a reviewed line in the check's configuration.
- **AC-180** `[crit]`: `GET /api/audit` requires `audit:read` — owner and office under the default role set. Worker and bookkeeper receive `403 NOT_PERMITTED`. The surface has no reachability narrowing: both roles that hold `audit:read` are unscoped for entity reachability (only the destructive-action narrowing below applies, and that narrows office).
- **AC-182** `[crit]`: An `audit_log` entry whose `action` is `purge` (any entity type), `delete` on `entityType = 'user'`, or `update` on `entityType = 'user'` where the payload diff touches `roles`, is returned by `GET /api/audit` only to callers whose destructive-action scope admits them — owner under the default role set (see [api.md §14.2.8](api.md#1428-audit-log)). For office the destructive-action scope excludes these entries from list and get-by-id responses alike. Permissions remain coarse — `audit:read` admits the caller to the surface; the destructive-action narrowing is orthogonal (parallel to the worker-scope pattern in [ADR-0019](../adr/0019-worker-data-scoping-repository-layer-predicate.md)).
- **AC-183** `[crit]`: A subscriber of the post-commit notification publisher that throws an unhandled exception does not roll back the originating mutation. The domain-state change and the `audit_log` row are committed; the publisher emits a structured operational log line with fields `event = 'audit-publisher-handler-error'`, `audit_entry_id` (the committed row's id), and `error_message`. Rationale: coupling subscriber correctness to domain transactionality would make third-party or future integrations a single point of failure for every mutation.
- **AC-184** `[crit]`: A scheduled cleanup job removes `audit_log` entries older than the configured retention window **[C]** (90 days default, see [architecture.md §12.2](architecture.md#122-company-configurable-settings)). Each run emits exactly one structured operational log line at `info` level with fields `event = 'audit-retention-cleanup'`, `window_days` (integer — the applied retention window), `removed_count` (non-negative integer; `0` on a no-op run), and `ran_at` (ISO 8601 run timestamp). Operators verify retention by querying the operational log for `event = 'audit-retention-cleanup'`; the Aktivität UI does not surface retention events. No `audit_log` row is produced by the cleanup job itself (scope is domain entities only — [data-model.md §5.10](data-model.md#510-audit-log-entity), [§6.10](data-model.md#610-audit-log-retention)).
- **AC-185** `[vis]`: Both activity surfaces — the project detail panel's activity feed and the global Aktivität view ([ui/management.md §8.13](ui/management.md#813-audit-view)) — render `audit_log` entries in reverse-chronological order (newest first, `id` as stable tiebreaker) and display `"Keine Aktivität"` when the result set is empty. The per-project feed paginates via `"Ältere anzeigen"` (older entries append without collapsing the current set); the global view paginates at the configurable list page size **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings)) and AND-composes its filter bar (entity type, entity, actor, date range, action) with a client-side block on `to < from`.
- **AC-186** `[vis]`: The payload drawer (`Details anzeigen`) is rendered on a feed entry only when the API response for that entry includes a non-null `payload`. Owner and office receive a payload on every entry; the drawer therefore renders on every feed row with a non-empty diff and expands inline without a route change.
- **AC-187** `[crit]`: The owner sees destructive-action entries (`purge`, `delete` on user, role-change updates) in the global Aktivität view — destructive-action scope admits owner with no filter. Office is the only other role that holds `audit:read` under the default matrix; the same scope excludes office from these entries. Verified by listing and get-by-id against the audit endpoint. (The per-project detail-panel feed filters on `{ancestorType: 'project', ancestorId}` ([architecture.md §11.12](architecture.md#1112-audit-ancestor-link)); of the destructive set only `purge` has a project ancestor, and a purged project leaves no openable detail panel — so the per-project surface has no observable destructive-visibility contract to pin.)
- **AC-188** `[crit]`: `audit_log.entityLabel` is a point-in-time snapshot captured at write time. Mutations on the target entity do not update prior audit rows' `entityLabel`: a rename writes a new row carrying the new label while prior rows keep the old one, and a delete/purge leaves prior rows' labels intact. Rationale: the activity feed must remain readable after renames or purges; a read-time JOIN or a back-fill would rewrite history — the exact misleading-state failure mode `entityLabel` exists to prevent ([data-model.md §5.10](data-model.md#510-audit-log-entity) "Entity label snapshot").

### 15.26 Attachments

Criteria for per-project file attachments ([data-model.md §5.13](data-model.md#513-attachment), [api.md §14.2.11](api.md#14211-attachments), [ui/project-detail.md §8.15](ui/project-detail.md#815-project-detail-page)).

- **AC-245** `[crit]`: `POST /api/projects/:id/attachments/init` creates exactly one `attachment` row with `status = 'pending'`, persists the per-blob wrapped-DEK envelope (`wrappedDek`, and `wrappedThumbDek` when `hasThumbnail = true`) by wrapping the client-supplied `dekMaterial` against the operator's binary `age` recipient, persists `wrappedDekVersion = 1` on the row alongside the envelopes (the format discriminator pinned by [ADR-0024](../adr/0024-binary-attachment-e2e-encryption.md) — same value applies to both `wrappedDek` and `wrappedThumbDek`, written once and never rewritten), and returns presigned-PUT descriptors that bind via SigV4: exact `originalKey` (URL path), exact `thumbKey` (when `hasThumbnail = true`), `Content-Type == 'application/octet-stream'` (sentinel — the row's plaintext `mimeType` is NOT signed into the PUT), `Content-Length` matching the declared `ciphertextSizeBytes` (and `ciphertextThumbSizeBytes` for the thumbnail), and `Content-MD5` matching the ciphertext body's RFC 1864 base64 MD5 (`ciphertextContentMd5` for the original; `ciphertextThumbContentMd5` for the thumbnail). The plaintext `sizeBytes` is also persisted on the row but is NOT signed into the PUT — it backs the per-file cap and rides the export envelope only. A request carrying a `mimeType` outside the whitelist ([data-model.md §5.13](data-model.md#513-attachment)), a `label` outside the enum, `sizeBytes` exceeding the per-file cap **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings)), `dekMaterial` malformed (not 32 bytes after base64-decode), `ciphertextSizeBytes` non-positive, a malformed `ciphertextContentMd5` — or, when `hasThumbnail = true`, missing/malformed `thumbDekMaterial`, `ciphertextThumbSizeBytes` non-positive or exceeding the per-thumb cap **[C]**, or a missing/malformed `ciphertextThumbContentMd5` — or a `fileName` violating the sanitization rules — empty, longer than 255 characters, containing a control character (`\x00`–`\x1F`, `\x7F`), a path separator (`/`, `\`), or a double-quote (`"`) — returns `422 VALIDATION_ERROR` and persists no row.
- **AC-212** `[crit]`: `POST /api/projects/:id/attachments/:attId/complete` flips the row to `status = 'ready'` only when a HEAD against object storage confirms both objects exist (when a thumbnail was requested) AND the reported size matches the row's persisted `ciphertextSizeBytes` (and `ciphertextThumbSizeBytes` when present) AND the reported content type equals the sentinel `'application/octet-stream'` for both blobs (the original AND the thumbnail — no `startsWith('image/')` carve-out, ciphertext is opaque). Any mismatch returns `409 CONFLICT` and leaves the row at `pending`. Complete called on an id already at `status = 'ready'` returns `409 CONFLICT` with no state change; complete called on an id the reaper has already removed returns `404 NOT_FOUND`. Defence-in-depth rationale: SigV4 binds the PUT to `Content-Length == ciphertextSizeBytes` and `Content-Type == 'application/octet-stream'`; HEAD re-asserts the same two properties at complete-time so a SigV4 bypass cannot leave the row finalized over a divergent stored object ([ADR-0024](../adr/0024-binary-attachment-e2e-encryption.md)).
- **AC-213** `[crit]`: An `attachment` row stuck at `status = 'pending'` longer than the configured orphan-reaper TTL **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings)) is removed by the scheduled reaper ([data-model.md §6.11](data-model.md#611-attachment-orphan-reaper)) together with a delete of `originalKey` and (when present) `thumbKey` in object storage. Each run emits exactly one structured operational log line at `info` level with fields `event = 'attachment-orphan-reaper'`, `ttl_minutes`, `removed_count` (non-negative, `0` on a no-op run), and `ran_at` (ISO 8601). No `audit_log` row is produced by the reaper. An in-flight sweep is drained on graceful shutdown before the database pool closes.

The response carries no `Cache-Control` directive (the protocol is non-cacheable per the WHATWG SSE spec). A subscribed connection receives every event the server broadcasts over the lifetime of the connection — verified by emitting a synthetic `storage_usage_changed` and asserting the connected client observes the event frame.
- **AC-269** `[crit]`: A connection held idle for the configured heartbeat window observes a heartbeat comment line (`:` keepalive) within 25 seconds — the interval is configurable **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings); default 25 seconds). When the server closes the connection, the client's browser-native `EventSource` reconnects with its own backoff and resumes receiving subsequent broadcasts; no `Last-Event-ID` replay is performed (events are invalidation hints, not a log — [ADR-0025](../adr/0025-realtime-ui-invalidation-via-sse.md)).
- **AC-270** `[crit]`: Each of the three AttachmentService mutation paths (`completeUpload` for `pending → ready`, `hide` for `ready → hidden`, `restore` for `hidden → ready`), the hidden reaper (`attachment-hidden-reaper` per [data-model.md §6.12](data-model.md#612-attachment-hidden-reaper)), the project-purge cascade (`DELETE /api/projects/:id/purge` per [AC-155](#1512-project-management) when the purged project carried at least one `attachment` row contributing non-zero bytes — i.e. a `ready` or `hidden` row), and the customer-delete cascade (`DELETE /api/customers/:id` per [AC-91](#1511-customer-management) when at least one of the atomically purged archived projects carried such a row) emits exactly one `storage_usage_changed` event post-commit that reaches a subscribed `/api/events` connection. A purge / customer-delete whose cascade removes no byte-bearing rows (purged project had no attachments or only `pending` rows; customer had no archived projects, or all archived projects were empty) emits zero events — pending rows contribute zero on every counter so the global figure does not move. The orphan reaper ([data-model.md §6.11](data-model.md#611-attachment-orphan-reaper)) deletes only `pending` rows and emits zero events for the same reason. Emission is post-commit — a mutation that aborts inside the transaction emits no event ([architecture.md §11.13](architecture.md#1113-realtime-invalidation-channel)); a customer-delete that throws inside its single enclosing transaction emits nothing even when the throw lands after the per-project purge writes (those writes roll back with the transaction). A subscriber whose write fails (closed socket, slow consumer) does not affect the dispatch to other subscribers — the failure is logged and the offending subscriber is removed; the originating mutation has already committed and is unaffected ([§11.11](#1111-notification-publisher-and-dispatch) parity).
- **AC-275** `[crit]`: A held `/api/events` stream observes a server-initiated end and is removed from the bus's subscriber set within one heartbeat interval after the underlying session becomes invalid. The check covers all four invalidation paths uniformly: natural expiry (`session.expiresAt < now`), logout / reaper deletion of the session row, and admin lockout (`users.active = false`). Each path is verified by an integration test that opens a stream, mutates the session or user state directly, and asserts the bus subscriber count drops within the budget. DB re-validation cadence is independent of broadcast rate — a test with N concurrent connections + M broadcasts per minute produces O(N) re-validation queries per heartbeat, not O(N·M). Transient DB errors during re-validation skip the tick rather than disconnecting the subscriber. Bug history: [CWE-613](https://cwe.mitre.org/data/definitions/613.html) — without re-validation a logged-out, expired, or deactivated user's tab keeps receiving frames until the client itself disconnects; a payload-bearing event would leak.
- **AC-276** `[crit]`: Each of the ten project-mutation sites (`ProjectCrudService.createProject`, `ProjectCrudService.updateProject` — including the assigned-worker-only path that writes only `project_workers`, `ProjectCrudService.deleteProject` for soft-delete / archive, `ProjectCrudService.restoreProject` for un-archive, `ProjectCrudService.purgeProject` for the hard delete in [AC-155](#1512-project-management), `ProjectTransitionService.transitionForward` per [AC-5](#151-core), `ProjectTransitionService.transitionBackward` per [AC-6](#151-core), `ProjectDatesService.updateDates` per [AC-7](#151-core), the `CustomerService.deleteCustomer` cascade per [AC-91](#1511-customer-management) when at least one archived project exists under the customer, and the `ImportService.import` non-dry-run paths ([api.md §14.2.4](api.md#1424-unified-data-exchange)) — both the override branch (TRUNCATE + re-insert into a non-empty target) and the empty-target branch (insert into empty tables) — on a successful commit) emits exactly one `project_changed` event post-commit that reaches a subscribed `/api/events` connection. A `CustomerService.deleteCustomer` whose target carries zero archived projects emits zero `project_changed` events — the project rows visible to subscribers are unchanged. An `ImportService.import` dry-run emits zero events (the path is read-only). Failure-isolation, post-commit, and tx-abort posture mirror [AC-270](#1528-realtime-events) for `storage_usage_changed` — emission is post-commit only, a mutation that aborts inside the transaction emits no event, and a subscriber whose write fails does not affect the dispatch to others or the originating mutation. For paths that may co-emit `storage_usage_changed` (`ProjectCrudService.purgeProject`, `CustomerService.deleteCustomer`), the two events' emit gates are independent — `project_changed` fires whenever the project row(s) change, regardless of whether the cascade also moved byte-bearing attachment rows.
- **AC-277** `[crit]`: Multi-user invalidation — two actors on the same install: an office session (the always-open observer) and a one-shot owner-authenticated request context driving every mutation from a session distinct from the office observer. The owner advances a project (`+1` transition per [AC-5](#151-core)), archives a project ([AC-61](#1512-project-management)), restores an archived project, edits a project's planned dates ([AC-7](#151-core)), edits an assigned-worker set, and creates a new project ([AC-59](#1512-project-management)). No worker actor: workers do not hold `project:transition`, `project:create`, `project:update`, `project:delete`, or `project:dates` per the role table in [api.md §14.3](api.md#143-authorization-rules), so none of the [AC-276](#1528-realtime-events) project-mutation sites are worker-reachable; contrast with [AC-273](#1529-storage-usage-ui), where workers DO hold `attachment:write` / `attachment:hide` and a worker arm carries weight. This is the value test the realtime channel exists to deliver for project lifecycle — the always-open-observer gap that mount + `visibilitychange` + post-mutation refresh alone leave open ([ADR-0025](../adr/0025-realtime-ui-invalidation-via-sse.md)), parity with [AC-273](#1529-storage-usage-ui) for storage usage.

### 15.30 Invoices

Criteria for the invoice domain ([data-model.md §5.15](data-model.md#515-invoice-entity), [data-model.md §5.16](data-model.md#516-invoice-sequence-entity), [api.md §14.2.14](api.md#14214-invoice-operations), [architecture.md §11.14](architecture.md#1114-invoice-domain), [ADR-0026](../adr/0026-invoices-immutability-and-zugferd.md)).

- **AC-285** `[crit]`: `POST /api/invoices` creates exactly one `Invoice` row with `status = 'draft'`, `number` null, `issueDate` null, `renderedPdfBinaryDescriptorId` null. The row's `projectId` is FK-bound to the supplied project; `taxMode` defaults to the live `company_profile.defaultTaxMode` when not supplied; `recipient` is pre-filled from the project's customer when not supplied. The endpoint requires `invoice:write`; callers without it receive `403 NOT_PERMITTED`. A draft on an archived project is rejected with `404 NOT_FOUND` (mirrors [AC-95](#1517-data-integrity)). The mutation produces one `audit_log` row via `mutate()` with `entityType = 'invoice'`, `action = 'create'`, ancestor `('project', projectId)` per [AC-177](#1523-audit-log).
- **AC-286** `[crit]`: `PATCH /api/invoices/:id` on a `status = 'draft'` row updates the specified fields (PATCH semantics — omitted fields unchanged); `totals` is re-derived server-side on every successful PATCH from `lines` + `taxMode`. `PATCH` on a row whose `status` is `'issued'` or `'cancelled'` is rejected with `422 VALIDATION_ERROR` and `code = INVOICE_FROZEN`; no field is mutated, no audit row is written. `DELETE /api/invoices/:id` on a `status = 'draft'` row hard-deletes the row; on a non-draft row it is rejected with `INVOICE_FROZEN`. Both mutations require `invoice:write`.
- **AC-287** `[crit]`: `POST /api/invoices/:id/issue` runs the issuance transaction ([architecture.md §11.14](architecture.md#1114-invoice-domain)) inside one DB transaction: allocates `number` from `invoice_sequence` via an atomic `UPDATE … RETURNING next_value` on `(year, 'invoice')` (row-exclusive lock equivalent to `SELECT FOR UPDATE`), snapshots `issuer` / `recipient` / `lines` / `taxMode` / `profile` onto the row, computes `totals` server-side, renders the PDF/A-3 + factur-x.xml, writes the binary descriptor, flips the parent project's `status` to `abgerechnet`, writes exactly one `audit_log` row with `action = 'invoice:issue'`, and commits. A failure at any step rolls back the entire transaction — the sequence increment never persists (gapless property), no binary is left orphaned, the project status stays at `rechnung_faellig`, and the draft remains editable. Post-commit, exactly one `invoice_changed` SSE event AND one `project_changed` SSE event reach subscribed `/api/events` connections (the project status changed, so both consumer surfaces must invalidate). Requires `invoice:write`.
- **AC-288** `[crit]`: Two concurrent issue calls on two distinct drafts within the same calendar year produce two invoices with sequential, gapless `number` values from the `(year, 'invoice')` sequence — never duplicates, never gaps. An issue call whose transaction rolls back after sequence allocation (render failure, binary write failure, project-status flip failure) returns the value to the sequence: a subsequent successful issue allocates the same value. Verified by an integration test driving the rollback path and asserting the next successful issue receives the rolled-back value.
- **AC-289** `[crit]`: `POST /api/invoices/:id/issue` rejects the call when the parent project's `status` is not `rechnung_faellig` with `409 CONFLICT` and `code = INVOICE_PROJECT_STATE`; no state change; no `audit_log` row; no SSE event. `POST /api/invoices/:id/issue` rejects with `422 VALIDATION_ERROR` when any of the following pre-issuance requirements is unmet: `performanceDate` is null, `lines` is empty, `recipient.name` is empty, `recipient.address.street` / `recipient.address.zip` / `recipient.address.city` (all three required per §14 UStG) is empty, or — when `taxMode = 'reverse_charge'` — `recipient.ustId` is empty. `POST /api/invoices/:id/issue` rejects when the `company_profile` singleton lacks any field required for the requested `taxMode` (e.g. `ustId` for `'standard'` or `'reverse_charge'`; `companyName` / `address` / `taxId` for every mode) with `422 VALIDATION_ERROR` and `code = COMPANY_PROFILE_REQUIRED`; `details.missingFields` enumerates the offending field paths. None of these failures persist a number allocation — the row-exclusive lock either is never acquired (pre-validation) or releases without an increment (validation inside the transaction).
- **AC-290** `[crit]`: `POST /api/invoices/:id/cancel` on a `status = 'issued'` row creates a new sibling row with `cancellationOf = <original.id>`, `number` allocated as `ST-YYYY-NNNN` from `invoice_sequence` `(year, 'storno')`, `lines` carrying sign-flipped `unitPrice` and `lineTotal`, `totals` re-derived from the negated lines, `issuer` / `recipient` / `taxMode` / `profile` / `performanceDate` snapshotted from the original, optional `cancellationReason` from the request body. The original row's `status` flips to `'cancelled'`; all other fields on the original are unchanged (write-once at the persistence layer — [AC-294](#1530-invoices)). Both rows persist forever. Exactly two `audit_log` rows are written inside the same transaction (`invoice:cancel` on the original, `invoice:issue` on the Storno). The parent project's `status` is **not** modified. Post-commit, exactly one `invoice_changed` SSE event reaches subscribers. Requires `invoice:write`.
- **AC-291** `[crit]`: `POST /api/invoices/:id/cancel` on a `status = 'draft'` row is rejected with `409 CONFLICT` and `code = INVOICE_NOT_ISSUED`. `POST /api/invoices/:id/cancel` on a `status = 'cancelled'` row is rejected with `409 CONFLICT` and `code = INVOICE_ALREADY_CANCELLED`. Neither rejection persists a number allocation, an audit row, or an SSE event.
- **AC-292** `[crit]`: A rendered invoice carries the legally-required boilerplate matching its snapshotted `taxMode` ([architecture.md §11.14](architecture.md#1114-invoice-domain)): `'standard'` renders per-line VAT columns plus the per-rate breakdown in totals; `'kleinunternehmer'` omits VAT columns and includes the `"Gemäß § 19 UStG wird keine Umsatzsteuer berechnet"` boilerplate (or the configured equivalent — German UI strings `[C]`); `'reverse_charge'` omits VAT on lines and includes the `"Steuerschuldnerschaft des Leistungsempfängers gemäß § 13b UStG"` boilerplate. Verified by an integration test that issues one invoice under each mode against the same draft template and asserts the rendered PDF/A-3's extractable text contains the expected boilerplate for the mode and not the boilerplate of the other modes.
- **AC-293** `[crit]`: An issued invoice's `profile` is `'zugferd-en16931'` (v1 — only valid value) and the rendered binary is a PDF/A-3 carrying an embedded `factur-x.xml` payload conforming to the EN 16931 Comfort profile. Verified by an integration test that drives the issuance path against a fixture draft, extracts the embedded XML from the resulting PDF/A-3, and asserts the XML validates against the EN 16931 schema (no XSD-validation errors). `profile` is snapshotted at issuance from the company-wide profile selection (v1: always `'zugferd-en16931'`); a future profile addition (e.g. `'xrechnung'`) lands by extending the renderer dispatch without schema migration ([ADR-0026 §E-invoice format](../adr/0026-invoices-immutability-and-zugferd.md#e-invoice-format)).
- **AC-294** `[crit]`: A direct `UPDATE` against an `invoices` row whose `status` is `'issued'` (touching any column other than the cancellation-flip path on `status`) is rejected. The application API never reaches this rejection — the route layer returns `INVOICE_FROZEN` first ([AC-286](#1530-invoices)) — but the persistence layer is the defense-in-depth backstop on direct DB writes that bypass the route layer (seed scripts, migrations, manual SQL). Mechanism is implementation-defined (a trigger, a column-level constraint, or a service-layer invariant exercised by direct-DB integration tests). Mirrors the project-status integrity posture in [AC-96](#1517-data-integrity).
- **AC-295** `[crit]`: A direct INSERT into `invoices` with a `number` that violates the regex `^(RE|ST)-\d{4}-\d{4,}$` (e.g. `RE-2026-1`, `re-2026-0001`, `RE-26-0001`, `XX-2026-0001`) is rejected by the database CHECK constraint. The application API never produces such a number — the issuance service formats from `invoice_sequence` directly — but the CHECK is the defense-in-depth backstop. Mirrors [AC-96](#1517-data-integrity) for project status.
- **AC-296** `[infra]`: The `INVOICE_OBJECT_LOCK_DAYS` env var ([architecture.md §12.2](architecture.md#122-company-configurable-settings)) is enforced by the boot-time bucket-safety probe ([architecture.md §11.4](architecture.md#114-object-storage-module)): the probe asserts the bucket's default-retention envelope on the invoice key prefix is **greater than or equal to** the env value. The invariant is data-integrity asymmetric — under-retention is the failure mode (bucket retention shorter than the env value means binaries become deletable before the legal horizon elapses); over-retention is operationally fine (the env declares the minimum; a longer bucket envelope still satisfies it). A bucket retention below the env value (env says 3650, bucket says 365) refuses to start the `app` service; a bucket retention above the env value (env says 3650, bucket says 7300) is accepted. `INVOICE_OBJECT_LOCK_DAYS = 0` disables the assertion entirely — the dev default, intended only for environments where the binaries are disposable. The probe runs alongside the existing attachment retention assertion (`STORAGE_OBJECT_LOCK_DAYS`) without merging the two — invoices and attachments have independent retention horizons.
- **AC-297** `[crit]`: `GET /api/invoices`, `GET /api/invoices/:id`, `POST /api/invoices`, `PATCH /api/invoices/:id`, `DELETE /api/invoices/:id`, `POST /api/invoices/:id/issue`, `POST /api/invoices/:id/cancel`, `GET /api/invoices/:id/pdf` all reject unauthenticated requests with `401 UNAUTHENTICATED` / `SESSION_EXPIRED`. Permission gates: read endpoints require `invoice:read` (owner / office / bookkeeper hold it); write / issue / cancel endpoints require `invoice:write` (owner / office). The worker role holds neither and receives `403 NOT_PERMITTED` on all eight endpoints. Verified by an API integration test that walks each role × each endpoint and asserts the expected status code.
- **AC-298** `[crit]`: Worker exclusion is enforced at the repository-predicate layer — the worker scope predicate ([ADR-0019](../adr/0019-worker-data-scoping-repository-layer-predicate.md)) returns the empty set for invoice rows on the worker role, so `GET /api/invoices` for a worker returns an empty list regardless of project assignments. A worker hitting `GET /api/invoices/:id` for an existing invoice returns `403 NOT_PERMITTED` (the row exists but the role is excluded); a non-existent invoice id returns `404 NOT_FOUND` — the three-way result mirrors the attachment policy ([AC-217](#1526-attachments)).
- **AC-299** `[crit]`: `GET /api/invoices/:id/pdf` returns the rendered PDF/A-3 + factur-x.xml for `status ∈ {'issued', 'cancelled'}` (or, per the implementation-defined transport in [api.md §14.2.14](api.md#14214-invoice-operations), enough information for the client to fetch and decrypt the rendered bytes through the synthetic-origin Service Worker path). A call on a `status = 'draft'` row is rejected with `409 CONFLICT` and `code = INVOICE_NOT_ISSUED`. Requires `invoice:read`; out-of-scope callers receive `403 NOT_PERMITTED` per [AC-298](#1530-invoices).

The rendered binary descriptor on the row continues to point at the bytes produced at issuance; no re-render path is exposed by the spec.
- **AC-305** `[crit]`: Issuing an invoice when the singleton `company_profile` row lacks any field required for the requested `taxMode` is rejected `422 VALIDATION_ERROR` with `code = COMPANY_PROFILE_REQUIRED` and `details.missingFields` enumerating the missing field paths. The check runs in the issuance transaction, after the sequence allocation lock is acquired but before the increment is committed — a rejection releases the lock without persisting the increment (gapless property is preserved). No `audit_log` row is written for the failed issue.

The same call against a non-archived project with zero invoices returns `409 CONFLICT` (regression coverage with AC-156, distinct error code). Pins [AC-308](#1512-project-management).

### 16.3 E2E Tests

The end-to-end path is covered by focused, isolated test files rather than a single monolithic scenario. This improves test isolation, failure diagnostics, and net-zero teardown. The steps below define the required behavioral coverage; the implementation may split them across multiple test files and specs.

**Smoke (login/logout cycle)**:

2. User enters credentials and logs in — Kanban view is displayed with 9 columns.

**Management flows (project, customer, user CRUD)**:

## 17. Risks and Mitigations

| Risk                                               | Impact                             | Mitigation                                                                                                                                                    |
| -------------------------------------------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 9 Kanban columns too tight on screen               | Usability                          | Horizontal scroll; responsive collapse tiers ([ui/behavior.md §10](ui/behavior.md#10-responsive-behavior)).                                                   |
| Over-styling action columns                        | Defeats Kanban principle           | Trust board structure; resist decorative urgency.                                                                                                             |
| API latency makes transitions feel sluggish        | Interaction feels unresponsive     | Optimistic UI updates + sub-300ms API target ([architecture.md §13.2](architecture.md#132-performance)).                                                      |
| Session management edge cases                      | User loses work or sees stale data | Sessions are checked on every API call. Expiry redirects to login cleanly. All mutations are immediate (no local drafts to lose).                             |
| Seed data dates become stale over time             | Demo loses impact                  | Dates are relative to deployment date ([data-model.md §7.5](data-model.md#75-date-range)). A re-seed operation refreshes them.                                |
| Soft-deleted projects consume storage indefinitely | Database growth                    | Archived projects are purged when their customer is deleted ([ADR-0017](../adr/0017-soft-delete-as-board-archive.md)). Customer deletion is the cleanup path. |
