# API Specification

---

## 14. API Specification

The API is the boundary between the front end and all persistent state ([architecture.md §11.1](architecture.md#111-mandatory-constraints)). The front end's state layer is a client of this API (see [architecture.md §11.3](architecture.md#113-state-layer-behavioral-contract)).

### 14.1 Design Principles

- **Single source of truth.** The API is authoritative for all persisted data.
- **Stack-agnostic contract.** Operations, inputs, and outputs — not transport, URLs, or serialization.
- **Consistent error representation.** Machine-readable code + human-readable message (see [§14.4](#144-error-handling)).
- **Stateless requests.** Each request carries all context needed, including authentication.
- **Pagination-ready.** List operations accept `offset`/`limit`. The contract must support pagination even if the current data volume does not require it.
- **Stable list order.** List operations return rows in a deterministic order — a second fetch yields the same sequence, and pagination pages do not overlap or skip rows.
- **Timestamps managed server-side.** `createdAt`, `updatedAt`, `statusChangedAt` are set by the server, never by clients.

Design notes:

- Failed login returns a generic error — no distinction between "user not found" and "wrong password".
- User profiles never include `passwordHash`.
- **Update self** is restricted to the caller's own record. The endpoint cannot affect any other user and does not accept a user ID. Identity-bearing fields (`username`, `roles`, `active`) are never updatable through this operation — those remain administrative (see [§14.2.3](#1423-user-management)).
- **Update self error paths** (mapped to the categories in [§14.4.1](#1441-error-categories)): missing or invalid session → authentication error; request body violating the allowed value set (e.g., `themePreference` outside `'light' | 'dark' | 'system'`, `pushMuted` not a boolean) → validation error.

**Project list filter parameters:** `status` (single or multiple), `customerId`, `plannedStartFrom`/`plannedStartTo` (date range), `hasNoDates` (boolean), `includeArchived` (boolean; default false; when true, includes `deleted = true` rows with archive state preserved), `search` (free-text across number, title, and customer name). Filters use AND logic. These filters serve both the project management view and the export operation.

Design notes:

- **Transitions are explicit operations**, not generic field updates — workflow adjacency rules are enforceable server-side.
- **Transition optimistic concurrency.** The client sends `expectedStatus` (the status it observed in its last read). The server advances only if the stored status still matches, otherwise the request is rejected with a `CONFLICT` error (see [§14.4](#144-error-handling)) — the client should refetch and present the current state. This deterministically prevents double-advance from sequential double-clicks, two tabs, or retried requests (see [data-model.md §6.4](data-model.md#64-concurrency) and [AC-94](verification.md#1517-data-integrity)).
- **Project number is immutable** after creation.
- **Soft delete**: archived projects are excluded from **list** results (the board and project-management view), but **Get project** returns the archived row as `200 OK` with `deleted: true` so the caller can render a read-only preview (e.g. from a deep link in the activity feed). Both reads respect caller scope — an archived project the caller was never allowed to see still surfaces as `404 NOT_FOUND`. Mutations on archived rows reject as `404 NOT_FOUND` (AC-95) regardless of read visibility, except for the dedicated **Restore project** path which flips `deleted = false` and returns the now-active row.
- **Purge is gated by archive-first.** A fresh project cannot be hard-deleted in a single step; the client must archive it (soft-delete via `DELETE /api/projects/:id`) before calling purge. This matches the trash-bin mental model and makes accidental destruction expensive (see [ADR-0017](../adr/0017-soft-delete-as-board-archive.md)).
- **Full project object returned** after every mutation so the client can reconcile without a separate fetch.
- **Update-dates** treats `plannedStart` and `plannedEnd` as a coordinated update. Clearing `plannedStart` also clears `plannedEnd` (see [data-model.md §6.8](data-model.md#68-date-validation)). `plannedEnd` before `plannedStart` is rejected.
- **Concurrent edit handling**: see [data-model.md §6.4](data-model.md#64-concurrency).
- The project response nests the full Customer object for rendering convenience.
- **Client-supplied `id` is optional.** When omitted, the server generates one. When supplied, it must be a syntactically valid id or the request is rejected as a validation error.
- **Idempotent replay.** If a row with the supplied id already exists AND every user-supplied field in the request body matches the stored row, the existing row is returned with the same success status as a fresh create. Replays do not create duplicates.
- **Field comparison for replay.** Participating fields: `number`, `title`, `customerId`, `status`, `plannedStart`, `plannedEnd`, `assignedWorkerIds`, `estimatedValue`, `notes`. Rules: `null`, `undefined`, and missing are equivalent; `assignedWorkerIds` compares as a set (order-independent); `estimatedValue` is rounded to two decimals before comparison; `plannedStart` and `plannedEnd` compare as calendar dates.
- **Idempotency conflict.** If a row with the supplied id exists but any participating field differs, the request is rejected with a conflict error code `IDEMPOTENCY_CONFLICT` (see [§14.4](#144-error-handling)). The stored row is not modified.
- **Soft-deleted match.** If the matching row is soft-deleted (see [data-model.md §6.9](data-model.md#69-soft-deletes)), the replay returns the row in its current (deleted) state. Clients must inspect the response rather than assume the row is active.
- **`number` uniqueness is separate.** Creating a project whose `number` collides with another row's `number` (regardless of idempotency) is rejected with a conflict error code `CONFLICT` and a German message naming the conflicting number (see [§14.4](#144-error-handling)). This is distinct from `IDEMPOTENCY_CONFLICT`.

#### 14.2.4 Unified Data Exchange

A single export and a single restore-only import cover the business-data layer (customers, projects, project-worker assignments). Per-entity bulk endpoints do not exist. A companion descriptor surface accompanies the export so the browser can assemble a single zip carrying both `data.json` and every `status='ready'` attachment as plaintext bytes (the binary download mirrors the per-project bulk-fetch path under [ADR-0024](../adr/0024-binary-attachment-e2e-encryption.md) — the server never sees plaintext bulk). Rationale and the three-layer persistence model live in [ADR-0018](../adr/0018-data-persistence-and-recovery-layered-strategy.md); the envelope shape is defined in [data-model.md §5.8](data-model.md#58-export-envelope).

Each `entries[i]` is either a fetchable descriptor (carrying a fresh presigned-GET on the ciphertext object plus the unwrapped 32-byte DEK) or an error row (`error = 'DEK_UNWRAP_FAILED'`); the discriminator lets the client surface skipped rows in the post-export summary without losing track of which rows were affected. Thumbnails are not included — the export carries plaintext data, not in-app rendering. Ordering is ascending `(createdAt, id)`: a row with `(createdAt, id)` strictly greater than the cursor at page-composition time is surfaced in a later page; a row whose status changed away from `ready` after the cursor passed it is not retroactively re-included or re-excluded.
```typescript
interface BinaryDescriptor {
  attachmentId: string;
  projectId: string;
  projectNumber: string; // e.g. "2026-0042"; sanitised by the client before use as a path component
  projectTitle: string; // sanitised by the client before use as a path component
  fileName: string; // plaintext file name; sanitised by the client before use as a path component
  sizeBytes: number; // plaintext size — drives the in-flight bytes-done counter
  // Mutually exclusive: a descriptor either carries the fetch triple (originalUrl + originalDekMaterial + expiresAt)
  // OR carries the error tag — never both, never neither.
  originalUrl?: string; // presigned-GET on the ciphertext object
  originalDekMaterial?: string; // base64 of the unwrapped 32-byte DEK that decrypts originalUrl
  expiresAt?: string; // ISO 8601; URL is useless past this point
  error?: 'DEK_UNWRAP_FAILED'; // surfaces a per-row unwrap failure to the client; row is skipped by the export
}
```

Design notes:

- **Envelope shape** — `{ schema_version: int, exported_at: ISO 8601 string, customers: Customer[], projects: Project[], project_workers: { projectId, userId }[] }`. Each entity carries every persisted field including `id`, audit timestamps, and archive state (see [data-model.md §5.8](data-model.md#58-export-envelope)).
- **Dry-run mode** — the client signals dry-run via `?dry_run=true` on the import request. The server runs full validation against the envelope (schema version, intra-envelope referential integrity, missing-user references against the target database, row-level constraints), returns a preview describing what would be written, and performs no writes. A subsequent read reflects no state change. On the dry-run path both intra-envelope and missing-user-reference issues are evaluated regardless of intra-envelope state and surfaced together in the preview, so a single dry-run surfaces every blocker before commit.
- **Override flag** — the client signals override via `?override=true`. When the target is non-empty, override wipes existing customers, projects, and project-worker assignments atomically with the restore. Without the flag, a non-empty target rejects the request with a specific error code.
- **Confirmation phrase** — when `override=true` AND the target is non-empty, the request body MUST include `confirmation_phrase` equal to the configured phrase **[C]**. The server compares after trimming leading/trailing whitespace; comparison is case-sensitive. A missing or non-matching phrase is rejected with a specific error code and performs no writes. Dry-run requests and override-into-empty-target requests are exempt. The phrase is a destructive-action guard, not an authorization factor; the authoritative gate remains `data:restore`.
- **Missing user references** — the user-id fields carried in the envelope (`customers.createdBy`, `customers.updatedBy`, `projects.createdBy`, `projects.updatedBy`, `project_workers.userId`) are checked against the target database's `users` table during envelope validation. Null or missing values on the nullable audit fields (`createdBy`, `updatedBy` — see [data-model.md §5.1](data-model.md#51-project-entity), [§5.6](data-model.md#56-customer-entity), and the policy in [AC-98](verification.md#1517-data-integrity)) are not treated as references and do not trigger the check. A value is counted as a missing reference only when it is present in the envelope and absent from the target's `users` table. If any referenced user id is absent in the target, the import is rejected with error code `MISSING_USER_REFS` and performs no writes. The check runs on both the dry-run path (issues appear in the preview) and the commit path (request rejected), so an operator can detect the divergence under `dry_run=true` before committing to `override=true`. Rationale: the business-data layer is consistent with the layered persistence model (see [ADR-0018](../adr/0018-data-persistence-and-recovery-layered-strategy.md)). When the target's `users` table has diverged from the dump's user references (reseed, deleted worker, divergent installation), a clean fail tells the operator the right tool is the full DB restore (Layer 2), not the portable import. The error is not papered over by silently nulling the references.
- **Validation ordering** — on the commit path, intra-envelope referential integrity is reported first under `VALIDATION_ERROR`; the missing-user-reference check runs only when the envelope is intra-consistent, and `MISSING_USER_REFS` is raised only on an intra-consistent envelope — never both codes in one response. On the dry-run path this gate does not apply: both classes are evaluated and surfaced together in the preview regardless of intra-envelope state. Rationale: the two failures have distinct remediations (re-export the envelope vs. restore Layer 2 / reseed users), and on the commit path reporting them sequentially keeps each error response self-explanatory; the dry-run path remains the place to view both classes in parallel.
- **Atomic failure** — if any row fails validation during a non-dry-run import (with or without override), no state change is persisted. The request has no partial outcome. The missing-user-reference check participates in this guarantee: detection happens in envelope validation, before any wipe or insert runs.
- **Schema version** — a monotonic integer. Exports stamp the current value; imports compare and reject mismatches outright (see [ADR-0018 §Decision](../adr/0018-data-persistence-and-recovery-layered-strategy.md#decision)).
- **Users and sessions are not exchanged** through this surface. Admin bootstrap ([ADR-0010](../adr/0010-first-run-admin-bootstrap.md)) creates the first user; test seeding uses a direct-DB helper confined to the test layer.
- **Binary descriptors — cursor stability.** The page server encodes `(createdAt, id)` into the opaque `after` token; ordering is ascending on that pair. A row with `(createdAt, id)` strictly greater than the cursor at page-composition time is surfaced in a later page; a row whose status changed away from `ready` after the cursor passed it is not retroactively re-included or re-excluded. The iteration is a snapshot of the rows reachable at first-page request time, not a stop-the-world transaction. A first call has no cursor; a `nextCursor` of `null` signals the end of the iteration. Cursor format is internal to the server; the client treats the token as opaque.
- **Binary descriptors — fresh URLs per page.** Every page signs URLs against the configured presigned-URL TTL **[C]** at the moment the page is composed; descriptors from earlier pages do not stay valid indefinitely. A presigned URL whose `expiresAt` has passed before the client fetches the entry triggers ONE re-fetch of the affected page (the client re-issues the same `after` cursor); on a second expiry the entry is skipped — see Error paths below.
- **Binary descriptors — totalCount and totalSizeBytes.** Both are server-computed at first-page composition (a single aggregate query over `status='ready'` rows) and are informational. `totalCount` is the row count; `totalSizeBytes` is the summed plaintext `sizeBytes` of the same rows. The client surfaces both in the pre-flight dialog as a single up-front readout. Subsequent pages return the same values for stability; a row inserted, deleted, or status-changed during the iteration drifts the live state away from these values, but the values themselves do not change within the iteration.
- **Binary descriptors — no total cap.** The export-all path has no aggregate file-count cap and no aggregate byte cap. The per-project bulk-fetch caps in [§14.2.11](#14211-attachments) (the **[C]** Attachment bulk-fetch caps) do not apply to descriptors. Rationale: the per-project caps exist to bound mobile field-capture working sets where the device assembles a subset zip; export-all is an owner desktop workflow whose streaming-zip working set is bounded by the largest single file in flight, not by the aggregate. Pagination keeps the per-page response bounded; the per-page limit **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings)) is the only ceiling.
- **Binary descriptors — no audit event.** Listing descriptors writes no `audit_log` row, parity with the existing `Export` operation in the same section. The action is a read; the `audit_log` invariant ([AC-177](verification.md#1523-audit-log)) ties audit rows to domain-state changes, and the operator-level forensic signal for a full-data extraction is captured via the request-level access log, not via a synthetic audit row without a state change.
- **Binary descriptors — rate limiting.** No bespoke per-export rate limit. The per-page descriptor ceiling **[C]** bounds one page; protection against parallel page-draining by a single caller falls under the same standard request rate limit applied to other authenticated routes. Defense-in-depth — the surface is gated to `data:export` (owner/office), so the threat model is a privileged caller scripting the endpoint, not an unauthenticated flood. If a stronger guard is later required (per-session concurrency cap, per-export budget), it lands as an explicit `[C]` here.
- **Vollständiger Export — manifest accompanies the zip.** The browser-assembled zip ([ui/daten.md §8.11.3](ui/daten.md#8113-vollständiger-export)) carries a `manifest.json` at the root listing `data.json` and every attachment that landed in the zip with the SHA-256 of the bytes at that path. The manifest is computed client-side during decrypt; the server has no involvement. Rationale: end-to-end integrity at export time is implicit (AES-256-GCM authenticated decryption fails loudly on tampered ciphertext; zip CRCs cover the wire-write), but a long-lived takeout artifact needs an offline verification surface — `manifest.json` lets an external verifier re-derive SHA-256 of every zip entry and cross-check without re-running the export. The full contract is pinned in [AC-252](verification.md#1514-data-exchange).
- **Binary descriptors — error paths** (mapped to the categories in [§14.4.1](#1441-error-categories)):
  - Missing or expired session → `401 UNAUTHENTICATED` / `SESSION_EXPIRED`.
  - Caller lacks `data:export` → `403 NOT_PERMITTED`.
  - Malformed `after` cursor (not parseable as the documented opaque token) or `limit` outside the configured per-page bounds → `422 VALIDATION_ERROR`.
  - A wholesale failure to unwrap any envelope on the requested page (operator's binary `age` identity not loaded, defense-in-depth — the boot probe in [§14.2.11](#14211-attachments) blocks startup without it) → `500 SERVER_ERROR`. A per-row envelope-unwrap failure (corrupt envelope or partial key rotation on a single row, see `DEK_UNWRAP_FAILED` semantics in [§14.2.11](#14211-attachments) and [AC-244](verification.md#1526-attachments)) is surfaced inline as `entries[i].error = 'DEK_UNWRAP_FAILED'` with the fetch fields omitted; the page itself remains `200 OK`.
  - Mid-drain expiry: a presigned URL inside an `entries[i]` whose `expiresAt` has passed becomes useless. The client re-fetches the affected page (re-issues the same `after` cursor) once to obtain fresh URLs; on a second expiry of the same entry's URL the client logs and skips that entry. The endpoint does not reissue partial pages; cursor stability across re-fetches is provided by the deterministic `(createdAt, id)` ordering above.

#### 14.2.7 Backup Status

The Layer 2 backup-freshness badge ([verification.md §15.22](verification.md#1522-backup-and-recovery), [ADR-0020](../adr/0020-layer-2-encrypted-r2-backups-with-operator-loaded-drills.md)) is consumed only by the authenticated owner. The status fields piggyback on the two session-establishment responses, `POST /api/auth/login` and `GET /api/auth/me` (see [§14.2.1](#1421-authentication--user-profile)) — no dedicated route. The earlier unauthenticated `GET /api/backup/status` endpoint was removed together with the login-screen badge surface.

Design notes:

- **Owner-only payload.** Both `/api/auth/login` and `/api/auth/me` include the `backupStatus` field for callers with role `owner`; the field is omitted for every other role. Absence drives the client's "no badge on this surface" branch. The two session-establishment paths are kept symmetric on purpose — a log-out/log-in without this parity leaves the badge stuck on the last `/me`-hydrated value until a full reload, which is the class of staleness AC-171 rules out.
- **Unreachable source → field omitted, not silence.** Misleading state is a critical defect class ([ADR-0014](../adr/0014-ac-tier-system-critical-vs-design.md)). When the status row is unreachable, the server omits the field; the client maps `undefined` to the explicit "Status unbekannt" badge state ([AC-171](verification.md#1522-backup-and-recovery)).
- **No errors on absence of prior runs.** The pre-seed row from the baseline migration is always present. The service reports `lastBackupOk=false` with `lastBackupAt=null` for a DB that has never run a backup; the client derives the `backup-never-run` badge reason from that shape.

#### 14.2.8 Audit Log

A read-only surface over [`audit_log`](data-model.md#510-audit-log-entity). The table is append-only ([data-model.md §6.10](data-model.md#610-audit-log-retention)); no create, update, or delete operation is exposed.

| Operation              | Input                                                                                                                                           | Output                                 | Notes                                                                                                                                                                                                                                                             |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **List audit entries** | optional: offset, limit, filters: `entityType`, `entityId`, `ancestorType`, `ancestorId`, `entityLabelQuery`, `actorId`, `from`, `to`, `action` | audit entry list, total count          | Returns entries visible to the authenticated caller per the scoping rules below. Ordered by `createdAt` descending (newest first), stable tiebreaker `id`. Default page size matches other list endpoints. Requires `audit:read` permission. Filters AND-compose. |
| **Get audit entry**    | audit entry id                                                                                                                                  | single entry, or not-found / forbidden | Returns the full entry subject to the scoping rules below. Requires `audit:read` permission.                                                                                                                                                                      |

Design notes:

- **RBAC-sliced reads.** A single scope predicate — `auditDestructiveScopeForCaller(user)` — composes at the repository layer, following the pattern established in [ADR-0019](../adr/0019-worker-data-scoping-repository-layer-predicate.md) (permissions stay coarse; scope is orthogonal to capability). For owner the predicate returns null (no filter); for every other role that holds `audit:read` (office under the current matrix) it returns a `WHERE` fragment excluding destructive entries, defined as: `action = 'purge'` on any entity type; `action = 'delete'` AND `entityType = 'user'`; `action = 'update'` AND `entityType = 'user'` where the payload diff touches `roles`. The predicate AND-composes with the permission gate. An earlier revision also carried a reachability predicate for workers (self-authored + project-reachability); it was removed when workers lost `audit:read` — the audit surface is administrative, not worker-facing.
- **Three-way result (same as ADR-0019).** Get-by-id distinguishes three outcomes: not-found (`404 NOT_FOUND`), in-scope (`200`), out-of-scope (`403 NOT_PERMITTED` — only the destructive-action narrowing can produce this, and only for office). Out-of-scope must not be collapsed to not-found — the caller-identity boundary matters for the same reasons as in [AC-147](verification.md#1521-role-scoping).
- **Actor name in response.** For a `user`-actor entry, the response includes the acting user's `displayName` as a derived read-time convenience — independent of whether the caller holds `user:read`. The field is a presentation aid for the activity feed; it is not a substitute for the `user` list endpoint and carries no additional user-record fields. System-actor entries include `actorReason` instead.
- **Entity filters.** `entityId` takes a UUID and matches rows equal to that id. `ancestorType` + `ancestorId` take the same shape and match rows whose write-time ancestor pair equals `(ancestorType, ancestorId)` ([architecture.md §11.12](architecture.md#1112-audit-ancestor-link)) — this is the filter the project-detail activity feed uses so the one call returns project rows plus every nested-entity row (`project_worker`, `attachment`) scoped to that project. `entityLabelQuery` takes a case-insensitive substring of at least three characters and matches rows whose frozen `entityLabel` contains it — it powers the Aktivität filter bar, where the caller types a name. LIKE metacharacters are escaped (`%`, `_`, `\`) so user input is treated literally. Rows whose `entityLabel` is null (import and retention paths that cannot supply a label) are excluded from `entityLabelQuery` matches by construction. The substring match is backed by a pg_trgm GIN index on `entity_label`; sub-three-character queries are rejected (422) to keep the index useful. Ancestor filters are served by the compound index `audit_log_ancestor_idx`.
- **Payload drawer.** Entries include `payload` as returned. The client renders the diff drawer only on rows with a non-null, non-empty payload.
- **Pagination and ordering.** Entries are ordered newest-first by `createdAt`, with `id` as the deterministic tiebreaker. A second fetch with the same filters yields the same sequence; pages do not overlap or skip rows (per [§14.1](#141-design-principles)).
- **Error paths** (mapped to the categories in [§14.4.1](#1441-error-categories)):
  - Missing or expired session → `401 UNAUTHENTICATED` / `SESSION_EXPIRED` (authentication error).
  - Authenticated caller lacks `audit:read` → `403 NOT_PERMITTED` (authorization error).
  - Get-by-id on an existing row outside the caller's scope → `403 NOT_PERMITTED` (authorization error — see three-way result above).
  - Get-by-id on a non-existent row → `404 NOT_FOUND`.
  - Filter or pagination parameters failing schema validation → `422 VALIDATION_ERROR`.
  - `from`/`to` inverted (`to < from`) → `422 VALIDATION_ERROR`.
  - `entityType`, `ancestorType`, or `action` filter values outside the documented enum / vocabulary ([data-model.md §5.10](data-model.md#510-audit-log-entity)) → `422 VALIDATION_ERROR`.
  - Request using an HTTP method other than the documented reads on the audit endpoints → `405 METHOD_NOT_ALLOWED`.
- **Post-commit publisher contract.** The notification publisher module exposes:
  - `onAuditCommitted(handler)` — registers a handler invoked with the committed audit entry after the domain transaction commits. Handlers are invoked once per commit, in registration order.
  - `setOperationalLogger(logger)` — wires the structured operational logger used to surface handler failures.
    A throwing handler does not roll back the originating mutation — see [AC-183](verification.md#1523-audit-log) for the failure-surface contract. The publisher is process-local; restart-durability is out of scope (in-process dispatch per [ADR-0021](../adr/0021-audit-log-and-notifications-single-write-path.md)).

| **Get rule**    | rule id                                                      | single rule            | Full rule or not-found. Requires `notifications:manage`.                                                                                                                               |
| **Create rule** | eventClass, stateFilter?, recipientSpec, enabled             | created rule           | `recipientSpec` must carry at least one non-empty part. `stateFilter` permitted only on `project.transition_forward` / `project.transition_backward`. Requires `notifications:manage`. |

Design notes:

- **Validation rules** (all raise validation error on breach):
  - `eventClass` must be a member of [`NotificationEventClass`](data-model.md#511-notification-rule).
  - `stateFilter` must be null unless `eventClass` is a transition event; when present, it must be a member of the configured workflow-state set ([index.md §3](index.md#3-workflow-states)).
  - `recipientSpec.roles` must be a subset of the configured role set ([index.md §4.2](index.md#42-users)); `userIds` must each reference an active `UserAccount`; `includeAssignedWorkers` is rejected when `eventClass` is a non-project-scoped event (`backup.failed`, `disk.threshold_reached`).
  - At least one of `roles`, `includeAssignedWorkers`, or `userIds` must resolve to a non-empty part — an enabled rule with an empty spec is rejected.
- **Error paths** (mapped to the categories in [§14.4.1](#1441-error-categories)):
  - Missing or expired session → `401 UNAUTHENTICATED` / `SESSION_EXPIRED`.
  - Caller lacks `notifications:manage` → `403 NOT_PERMITTED`.
  - Get / update / delete on an unknown id → `404 NOT_FOUND`.
  - Validation breaches above → `422 VALIDATION_ERROR`.
- **Take-effect semantics.** A successful create, update, or delete affects the next event committed after the change; in-flight events use the rule set read at their own commit.
- **No uniqueness constraint on `(eventClass, stateFilter, recipientSpec)`.** Duplicates are permitted; the publisher's recipient-union dedups by `user_id` at resolution. A uniqueness constraint would force admins to reason about rule-equivalence — exactly the complexity the flat model avoids ([ADR-0023](../adr/0023-notification-rules-db-stored-closed-event-catalog.md)).

#### 14.2.10 Push Subscription

Browser push subscription management. Authenticated self-scope — a caller manages only their own subscriptions. The VAPID public-key read is the only unauthenticated route in this section (the key is intentionally public; the private half never leaves the server).

| Operation                 | Input                                     | Output                             | Notes                                                                                                                                                         |
| ------------------------- | ----------------------------------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Read VAPID public key** | —                                         | `{ vapidPublicKey: string\|null }` | Unauthenticated. Returns the server-derived VAPID public key, or `null` when VAPID credentials are not configured on the server. GET-only; non-GET verbs 405. |
| **Subscribe to push**     | endpoint, keys (p256dh, auth), userAgent? | success/failure                    | Registers a subscription for the authenticated caller. Re-subscribing an existing `endpoint` updates the row rather than duplicating.                         |
| **Unsubscribe from push** | endpoint (or subscription id)             | success/failure                    | Removes one of the caller's subscriptions. Unknown endpoints / ids are idempotent no-ops.                                                                     |

Design notes:

- **VAPID key is server-sourced.** The client fetches the public key at subscribe time from the read endpoint; the server derives it from the private half at startup, so the operator maintains a single server-side secret. Short HTTP cache (5 minutes) — a deploy rotates the key and the browser cache expires well inside one user session.
- **Ownership is derived.** The server sets `userId` from the session; a client-supplied user id is ignored.
- **Self-scoped lookups.** Subscribe / unsubscribe filter by `(subscription.id OR endpoint, userId = session.userId)` before acting. An id or endpoint belonging to another user is treated as unknown — the no-op response is indistinguishable from a truly-unknown one. Rationale: self-scope endpoints must not enumerate other users' subscriptions.
- **No auto-request.** The permission prompt is user-initiated from the settings affordance ([ui/index.md §8.7.2](ui/index.md#872-user-menu)); the server never pushes a registration on login.
- **Dispatch-time pruning.** A subscription the browser reports permanently dead at dispatch is removed server-side — housekeeping, not a user-facing operation.
- **Mute interaction.** `pushMuted = true` suppresses delivery to every subscription the user owns; rows are retained, so unmuting restores delivery without a re-subscribe.
- **Rate limiting.** Mutation paths (`POST`, `DELETE`) share a per-session mutation bucket ([architecture.md §12.2 rate-limit config](architecture.md#122-company-configurable-settings) `[C]`); exceeding returns `429 RATE_LIMITED`. The VAPID public-key read is not rate-limited (unauthenticated, GET-only, 5-minute cache).
- **Error paths** (mapped to the categories in [§14.4.1](#1441-error-categories)):
  - Missing or expired session → `401 UNAUTHENTICATED` / `SESSION_EXPIRED` (subscribe / unsubscribe only; the VAPID read is unauthenticated).
  - Malformed `endpoint` / `keys` → `422 VALIDATION_ERROR`.
  - `endpoint` exceeds 2048 characters, `keys.p256dh` or `keys.auth` exceeds 512 characters, or `endpoint` is not a syntactically valid URI → `422 VALIDATION_ERROR` (bounded to protect the subscription table from pathological input).

#### 14.2.11 Attachments

Per-project file attachments. The app never handles attachment bytes — uploads and downloads go directly between the client and object storage via server-issued presigned URLs. Bytes on storage are end-to-end encrypted per [ADR-0024](../adr/0024-binary-attachment-e2e-encryption.md): the browser AES-256-GCM-encrypts each blob with a per-attachment data-encryption key (DEK), the server wraps the DEK with the operator-loaded binary `age` recipient and persists only the wrapped envelope on the row, and storage objects carry the sentinel content type `application/octet-stream`. Each upload URL is a signed PUT that pins the exact object key, ciphertext content type, ciphertext content length, and ciphertext body MD5 via the SigV4 signed-headers list. Metadata is the `Attachment` entity ([data-model.md §5.13](data-model.md#513-attachment)).

```typescript
interface PresignedUpload {
  url: string; // PUT target — body goes here
  headers: Record<string, string>; // headers the client must send verbatim
  expiresAt: string; // ISO 8601 — after this the descriptor is useless
}

interface BulkFetchResponse {
  data: BulkFetchEntry[]; // one entry per requested id, order preserved
}

interface BulkFetchEntry {
  attachmentId: string;
  originalUrl: string; // presigned-GET for the encrypted original
  originalDekMaterial: string; // base64 of the unwrapped 32-byte DEK for the original
  ciphertextSizeBytes: number; // for client-side progress accounting
  thumbUrl?: string; // presigned-GET for the encrypted thumbnail; null for non-photo
  thumbDekMaterial?: string; // base64 of the unwrapped 32-byte DEK for the thumbnail; null for non-photo
  ciphertextThumbSizeBytes?: number; // null for non-photo
}
```

Design notes:

- **No byte traffic through the app.** Upload bytes travel client → object storage; download bytes travel object storage → client. The application sees only the metadata row, the wrapped DEK envelope it persisted at init, and the HEAD check at complete time. This is what lets the app stay small regardless of attachment volume, and it is what keeps the cleartext bytes off the VPS even after the e2e refactor (the server holds only the wrapping operation, never the plaintext bytes — [ADR-0024](../adr/0024-binary-attachment-e2e-encryption.md)).
- **Presigned-PUT signed headers — ciphertext binding.** Every init signs `Content-Type = 'application/octet-stream'` (the storage-object sentinel; the row's plaintext `mimeType` is never on the wire to storage), `Content-Length = ciphertextSizeBytes` (or `ciphertextThumbSizeBytes` for the thumbnail), and `Content-MD5 = ciphertextContentMd5` (RFC 1864 base64 of the ciphertext body's MD5; or `ciphertextThumbContentMd5`) into the URL via SigV4. The exact storage key is bound by the URL path. Any divergence on either the URL or the headers is rejected at signature verification; the storage provider additionally verifies `Content-MD5` against received bytes and rejects with `BadDigest` on mismatch. The plaintext-MIME and plaintext-`sizeBytes` fields are server-side row state (they back the download `Content-Disposition` and the per-file cap check at init); they are never sent on the wire to storage.
- **Two integrity layers, two failure modes.** `Content-MD5` is the storage-level body-match guarantee on PUT — it catches a body-substitution by a SigV4-bypassing attacker before the storage provider commits the object (see [ADR-0022](../adr/0022-binary-storage-b2-compliance-object-lock.md)). The AES-256-GCM authentication tag (verified on every decrypt; see Service Worker decrypt path below) is the cryptographic-integrity guarantee for the bytes — it catches any tampering after commit. Both layers stay; each catches a different class of failure. MD5 is not collision-resistant, so the PUT-side binding is "any bytes hashing to the signed MD5", not "exact bytes"; the AES-GCM tag is what makes the served bytes unforgeable on the read path. Under the current single-actor flow (the same client computes the MD5, calls init, encrypts, and PUTs the ciphertext) the MD5 distinction is academic; if a future flow ever splits init from PUT across actors, the spec must NOT continue to claim MD5 binds the URL to exact bytes. Why PUT over the AWS POST-policy form: B2's S3-compatible API does not implement browser-based POST uploads (B2 [returns 501 NotImplemented](https://www.backblaze.com/docs/cloud-storage-s3-compatible-api)); presigned PUT is the cross-provider lowest common denominator (AWS, B2, R2, MinIO, Wasabi).
- **Complete is retry-safe while the row is at `pending`.** A 409 from a HEAD miss leaves the row retry-eligible; a subsequent complete with both objects now HEAD-present succeeds. The one-way `pending → ready` transition (see [AC-212](verification.md#1526-attachments)) is what makes this safe: the row cannot be finalized twice, and a second complete on a `ready` row is itself rejected with 409.
- **Client-generated thumbnail.** For photo MIME types the client produces the thumbnail before calling init, AES-GCM-encrypts both blobs locally with their per-blob fresh DEKs, computes the ciphertext byte size + MD5 of each, and sends `hasThumbnail = true` plus `ciphertextThumbSizeBytes` + `ciphertextThumbContentMd5` + `thumbDekMaterial` (base64). The server wraps both DEKs (one per blob) and issues two presigned PUT descriptors, each pinning the exact ciphertext size + MD5 of its blob; the client PUTs both. The server does not transform bytes. The thumbnail format and the sizing parameters live in `ARCHITECTURE.md § Attachments Module` (impl-specific).
- **DEK material on the wire.** Init carries `dekMaterial` (and `thumbDekMaterial` when a thumbnail is uploaded) — base64 of a 32-byte AES-256-GCM key generated in the browser via `crypto.getRandomValues`, fresh per attachment. The server validates length after base64-decode and wraps with the operator's binary `age` recipient before persisting on the row as `wrappedDek` (and `wrappedThumbDek`); the unwrapped material is never persisted. Rationale and the rejected server-DEK alternative live in [ADR-0024](../adr/0024-binary-attachment-e2e-encryption.md).
- **Wrapped-envelope confidentiality boundary.** `wrappedDek` and `wrappedThumbDek` are the entire crypto perimeter on B2 ciphertext — a leaked DB dump must remain useless without the operator-loaded `age` identity. The wrapped envelope columns are NEVER returned in any list, get, trash-listing, restore, or complete response. Only the **unwrapped** DEK leaves the server boundary, and only on the per-request `Get download URL` and `Bulk fetch` surfaces above; the wrapped blob itself never crosses the API boundary in those directions. The single documented exception is `GET /api/export` ([§14.2.4](#1424-unified-data-exchange)) — the wrapped envelopes ride the export envelope so attachments restore decryptable post-import (see [AC-220](verification.md#1526-attachments)); the export envelope is therefore sensitive material with the same handling discipline as a DB dump. The columns are also schema-level audit-excluded so a future audited mutation cannot leak them; see [data-model.md §5.13 "Wrapped envelope is server-handed-back"](data-model.md#513-attachment) for the entity-side contract and the audit-exclusion mechanism in `ARCHITECTURE.md`.
- **Service Worker decrypt path.** The SPA installs a Service Worker that intercepts requests to a synthetic same-origin URL scheme `/encrypted-storage/<projectId>/<attachmentId>.<variant>` (where `variant ∈ {'original', 'thumbnail'}`). On each intercepted request the SW (1) calls the matching `Get download URL` endpoint to obtain `{ url, expiresAt, dekMaterial }`, (2) fetches `url` to obtain the ciphertext bytes from object storage, (3) AES-256-GCM-decrypts with `dekMaterial` (the AES-GCM authentication tag is verified on every decrypt and rejects tampered bytes), and (4) returns the plaintext bytes through the Fetch response. `<img src="/encrypted-storage/...">`, `<iframe src="/encrypted-storage/...">` for PDF preview, and `<a href="/encrypted-storage/..." download>` keep working unchanged — the synthetic origin is the integration seam. The DEK is held only for the lifetime of the SW-side decrypt; nothing on the row leaks to client storage. Full rationale and the rejected explicit-fetch-and-blob-URL alternative live in [ADR-0024](../adr/0024-binary-attachment-e2e-encryption.md).
- **Labels and MIME whitelist.** Init validates `label` against the closed enum ([data-model.md §5.13](data-model.md#513-attachment)) and the plaintext `mimeType` against the whitelist. Outside values produce a validation error with no row persisted.
- **Scoping — repository-layer predicate, parallel to [ADR-0019](../adr/0019-worker-data-scoping-repository-layer-predicate.md).** `attachmentScopeForCaller(user)` returns `null` for owner, office, bookkeeper; for worker it returns a `WHERE` fragment admitting only attachments whose project the worker appears in (`project_workers`). The predicate is AND-composed with the permission gate. Get-by-id, like the project get-by-id pattern, distinguishes three outcomes: `404 NOT_FOUND`, `200` in-scope, `403 NOT_PERMITTED` out-of-scope (the existence-through-role-boundary policy in [AC-147](verification.md#1521-role-scoping) applies identically).
- **Worker write rules.** A worker may call init + complete only on a project where the worker is in `project_workers`; otherwise the request is rejected with `403 NOT_PERMITTED`. A worker holds `attachment:hide` and may hide only attachments where `createdBy = callerId` AND `now - createdAt` is within the configured self-delete grace window **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings)). Outside this window, or for attachments the worker did not author, hide is rejected with `403 NOT_PERMITTED`. Owner and office hold `attachment:hide` without the author or time constraint. Worker does not hold `attachment:trash` — the Papierkorb (list-hidden + restore) is owner / office only.
- **Bookkeeper read-only.** Bookkeeper holds `attachment:read` (unscoped, same MVP stand-in as `project:read` and `customer:read`) but no write, hide, or trash. The same `attachment:read` admits both the `Get download URL` and the `Bulk fetch` surfaces — no new permission was introduced for the e2e refactor.
- **Archive interaction.** Hide and init are forbidden on archived projects (read-only-preview invariant — parity with [AC-95](verification.md#1517-data-integrity)); the route returns `404 NOT_FOUND` to mirror the project-mutation contract. List-trash and restore are permitted on archived projects so an operator can recover binaries before lifecycle reap consumes the hidden version.
- **Mutation auditing.** Init, hide, and restore go through the single-write-path helper and produce `audit_log` rows ([data-model.md §5.10](data-model.md#510-audit-log-entity), [AC-177](verification.md#1523-audit-log)). The action vocabulary adds `attachment:add`, `attachment:hide`, `attachment:restore` (init / user-DELETE / Papierkorb-restore respectively). Complete is a state-machine finalize on the same row init created; it does not produce an additional audit row — the `attachment:add` entry is the authoritative record of the attachment entering the project. `entityType` is `'attachment'` and `entityId` is the attachment's `id`; the write-time ancestor is `('project', projectId)` ([architecture.md §11.12](architecture.md#1112-audit-ancestor-link)) so the per-project activity feed's `ancestorType + ancestorId` filter picks the row up alongside project- and `project_worker`-scoped rows in one indexed query. `payload` retains `projectId` for parent-linkage rendering and never carries `wrappedDek` / `wrappedThumbDek` (schema-level exclusion). The orphan reaper ([data-model.md §6.11](data-model.md#611-attachment-orphan-reaper)) removes stuck `pending` rows together with their `originalKey` / `thumbKey` objects without producing audit rows — its targets never entered the user-visible domain.
- **Bulk-fetch filename collisions.** The browser-side streaming-zip assembler ([ADR-0024](../adr/0024-binary-attachment-e2e-encryption.md)) writes one entry per attachment using each row's `fileName`; on collisions the client deduplicates. The disambiguation strategy is a client-side concern (the contract carries `attachmentId` and `fileName` per entry — the client owns naming inside the archive).
- **Non-`ready` rows are ineligible for bulk fetch.** The 422 on a `pending` or `hidden` id (see Error paths below, and the bulk-fetch AC in verification.md §15.26) is a correctness guard, not an authorization check: a `pending` row's backing objects may not yet be durable (HEAD may still fail), and a `hidden` row carries a delete marker on the current version (the restore path is the way back to `ready`), so including either in a fetched batch risks a broken decrypt. On rejection the client must refetch the list and exclude any row no longer at `ready` before retrying.
- **Error paths** (mapped to the categories in [§14.4.1](#1441-error-categories)):
  - Missing or expired session → `401 UNAUTHENTICATED` / `SESSION_EXPIRED`.
  - Caller lacks the relevant permission → `403 NOT_PERMITTED`.
  - Caller out of scope for the project (worker path) → `403 NOT_PERMITTED` on a row that exists; `404 NOT_FOUND` on a missing id — distinguishable, see [AC-147](verification.md#1521-role-scoping).
  - Init: any of — `mimeType` outside whitelist, `label` outside enum, plaintext `sizeBytes` exceeds the configured per-file cap **[C]**, `fileName` violating the sanitization rules (empty, > 255 characters, contains a control character `\x00`–`\x1F`/`\x7F`, path separator `/`/`\`, or double-quote `"`), `dekMaterial` malformed (not 32 bytes after base64-decode), `ciphertextContentMd5` malformed (not RFC 1864 base64), `ciphertextSizeBytes` non-positive — and, when `hasThumbnail = true`, the analogous violations on `thumbDekMaterial`, `ciphertextThumbContentMd5`, `ciphertextThumbSizeBytes` → `422 VALIDATION_ERROR`. No row persisted. The double-quote ban is load-bearing: `fileName` is interpolated into the `Content-Disposition` header of presigned-GET download responses (see [data-model.md §5.13](data-model.md#513-attachment) "Filename sanitization"). The exact relationship between plaintext `sizeBytes` and `ciphertextSizeBytes` is implementation-defined (AES-GCM with prefixed nonce adds a fixed overhead, but the spec does not pin the precise number) — the contract is that the client declares the ciphertext size and the server signs the PUT against the declared figure.
  - Complete: HEAD finds an object missing, size mismatches the row's persisted `ciphertextSizeBytes` (or `ciphertextThumbSizeBytes`), or content type ≠ `application/octet-stream` → `409 CONFLICT`. The row stays `pending`; the orphan reaper ([data-model.md §6.11](data-model.md#611-attachment-orphan-reaper)) is the only path that removes it.
  - Complete on a row already at `status = 'ready'` → `409 CONFLICT`. No state change, no duplicate HEAD verification — the transition is one-way and non-replayable.
  - Complete on an id the reaper has already removed (row gone) → `404 NOT_FOUND`. The client discards the pending upload state and does not retry.
  - Presigned-PUT expiry mid-upload: if the upload does not complete within the presigned-URL TTL **[C]**, the storage provider rejects the PUT with a signature/time error. The app never observes the expired upload. The `pending` row stays in place; the orphan reaper ([data-model.md §6.11](data-model.md#611-attachment-orphan-reaper)) removes it once the row's age passes the reaper TTL. Retry is a fresh init call (new row, new presigned descriptors, fresh DEK); reusing the original `attachmentId` after presigned expiration is not supported.
  - Delete (soft-hide): row missing or wrong project → `404 NOT_FOUND`. Row not in `status = 'ready'` (already hidden, still pending, racing client) → `409 CONFLICT`. Project archived → `404 NOT_FOUND` (mirrors AC-95). Worker outside the self-delete grace, or worker on a row they did not author → `403 NOT_PERMITTED`. Storage hide failure is best-effort: the audit row commits and the row stays at `hidden`; the operational logger records the orphaned key for follow-up — lifecycle reaps regardless.
  - List trash: project missing → `404 NOT_FOUND`; caller out of scope → `403 NOT_PERMITTED`; missing `attachment:trash` → `403 NOT_PERMITTED`.
  - Restore: row missing or wrong project → `404 NOT_FOUND`. Row not in `status = 'hidden'` (already restored, still pending, racing client) → `409 CONFLICT`. CAS-loss inside the transaction → `409 CONFLICT` (transient — refetch and retry). Storage `copyFromVersion` failure → `500 SERVER_ERROR` (the surrounding transaction rolls back, the row stays at `hidden`, the audit row is not committed; the user retries cleanly). Integrity errors → `422 VALIDATION_ERROR` with messages identifying the row: `versionId` null on a `hidden` row (restore is structurally impossible — capture failed at complete-time), or `hasThumbnail = true` with `thumbVersionId` null (gallery preview would be permanently lost — surface the integrity violation rather than silently restore a broken row).
  - Bulk fetch: requested count or summed plaintext `sizeBytes` exceeds the configured bulk-fetch caps **[C]** → `422 VALIDATION_ERROR` with `code = BULK_LIMIT_EXCEEDED` and a `limits` field naming both caps (see [§14.4.1](#1441-error-categories)). Any id not belonging to the supplied project, OR any id referencing a row whose `status` is not `'ready'` (`pending`, `hidden`) → `422 VALIDATION_ERROR`. The whole batch is rejected; no partial-serve. A failure to unwrap any envelope in the batch → `500 SERVER_ERROR` (defense-in-depth — the boot probe ([ADR-0024](../adr/0024-binary-attachment-e2e-encryption.md)) blocks startup without the binary identity loaded, so this is not reachable in normal operation).
  - Download-URL: unknown variant → `422 VALIDATION_ERROR`. Thumbnail on a non-photo row → `422 VALIDATION_ERROR`. A per-row envelope unwrap failure (envelope bytes corrupt on this row, or this row's envelope was wrapped to a different recipient than the currently-loaded binary identity — most likely a partial key rotation in progress) → `422 VALIDATION_ERROR` with `code = DEK_UNWRAP_FAILED`. The Service Worker translates this code to the [AC-244](verification.md#1526-attachments) placeholder render path. A wholesale failure to unwrap because the operator's binary `age` identity is not loaded → `500 SERVER_ERROR`. Defense-in-depth — the boot probe ([ADR-0024](../adr/0024-binary-attachment-e2e-encryption.md)) blocks startup if the identity is absent, so this 500 should never reach a live request; if it ever does, the failure surfaces here rather than silently leaking ciphertext-without-key.
  - A request using an HTTP method other than the documented verbs on any attachment endpoint → `405 METHOD_NOT_ALLOWED`.

### 14.3 Authorization Rules

- All API operations require authentication (valid, active session).
- The system implements a basic role-based permission matrix. All authenticated, active users can view all projects (list, get) and change their own password. Other operations — including mutations, imports, and exports — require specific permissions granted by role:

| Role       | Permissions                                                                                                                                                                                                                                                                                                                                                  |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| owner      | project:read, project:transition, project:dates, project:create, project:update, project:delete, project:purge, customer:read, customer:write, customer:delete, user:read, user:manage, user:delete, data:export, data:restore, audit:read, notifications:manage, attachment:read, attachment:write, attachment:hide, attachment:trash, auth:change-password |
| office     | project:read, project:transition, project:dates, project:create, project:update, project:delete, customer:read, customer:write, user:read, data:export, audit:read, attachment:read, attachment:write, attachment:hide, attachment:trash, auth:change-password                                                                                               |
| worker     | project:read, customer:read, attachment:read, attachment:write, attachment:hide, auth:change-password                                                                                                                                                                                                                                                        |
| bookkeeper | project:read, customer:read, attachment:read, auth:change-password                                                                                                                                                                                                                                                                                           |

Design notes:

- **Data exchange permissions are distinct from entity read/write permissions.** `data:export` gates the unified export ([§14.2.4](#1424-unified-data-exchange)); `data:restore` gates the unified import. Granting `project:read` does not imply `data:export`.
- **`notifications:manage` is admin-only.** The rule-editor surface ([§14.2.9](#1429-notification-rules)) is admin-only (owner under the default matrix, per [ADR-0023](../adr/0023-notification-rules-db-stored-closed-event-catalog.md)); rule changes affect who-sees-what platform-wide, so this aligns with `user:manage`.
- **`data:restore` is owner-only.** The import destroys or replaces business data in a single transaction — a destructive operation, reserved to the most privileged role consistent with `customer:delete`, `user:delete`, and `project:purge`.
- **Audit read is a single permission; visibility is narrowed by orthogonal scope predicates at the repository layer.** `audit:read` admits a caller to the audit surface at all. Destructive-action visibility is not a second permission: it is the `auditDestructiveScopeForCaller` predicate defined in [§14.2.8](#1428-audit-log) — owner gets a null filter; every other role gets a `WHERE` fragment excluding destructive entries. This follows [ADR-0019](../adr/0019-worker-data-scoping-repository-layer-predicate.md)'s decision that permissions stay coarse and scope is orthogonal to capability — encoding scope in the permission catalog would conflate _capability_ with _extent_.
- The API must be designed so that scoping reads to a subset (e.g. "projects assigned to user X") is an additive query filter on the list operation, not a restructuring of the endpoint.
- **Holding `project:read` does not imply unscoped access to every project; holding `customer:read` does not imply unscoped access to every customer.** For workers, reads (list and get) are scoped by caller identity — restricted to projects where the worker is in `project_workers`, and to customers referenced by those projects. Owner and office remain unscoped. Exactly one `project:read` and one `customer:read` exist in the matrix — this is observable in the table above. Bookkeeper has unscoped project and customer read as an MVP placeholder until an invoice-oriented view is introduced (see [index.md §4.2](index.md#42-users)). See also [AC-145](verification.md#1521-role-scoping), [AC-146](verification.md#1521-role-scoping), [AC-147](verification.md#1521-role-scoping), [AC-148](verification.md#1521-role-scoping).
- **Attachment permissions mirror the scope-orthogonal pattern.** `attachment:read`, `attachment:write`, `attachment:hide`, and `attachment:trash` are coarse capabilities; visibility is narrowed by `attachmentScopeForCaller` at the repository layer ([§14.2.11](#14211-attachments)), following [ADR-0019](../adr/0019-worker-data-scoping-repository-layer-predicate.md). Worker write is additionally gated at the service layer on the project being in the worker's scope; worker hide is additionally gated on authorship plus a self-delete grace ([§14.2.11](#14211-attachments)). `attachment:trash` (Papierkorb listing + restore) is owner / office only — workers can hide their own uploads inside the grace window but do not browse or restore the trash. Owner and office remain unscoped, consistent with their project and customer reach.
- Authorization checks must be enforced server-side. Client-side UI hiding (e.g. hidden buttons) is a UX convenience, not a security measure (see [architecture.md §13.6](architecture.md#136-security)).

### 14.4 Error Handling

Every error response carries two components:

| Component                  | Purpose                               | Example                                                   |
| -------------------------- | ------------------------------------- | --------------------------------------------------------- |
| **Machine-readable code**  | Programmatic handling by the client   | `INVALID_CREDENTIALS`, `SESSION_EXPIRED`, `NOT_PERMITTED` |
| **Human-readable message** | Display to the user (German, **[C]**) | `"Anmeldung fehlgeschlagen"`, `"Sitzung abgelaufen"`      |

An optional third component, `details`, may carry structured validation information (e.g., field-level errors from schema validation). Clients should handle its absence gracefully.

#### 14.4.1 Error Categories

The API must distinguish the following error categories. Each category has distinct client-side handling (see [ui/behavior.md — Asynchronous Mutation Behavior](ui/behavior.md#95-asynchronous-mutation-behavior)).

| Category                 | Meaning                                                                                                                                                                                                                                   | Client behavior                                                                                                                                                |
| ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Authentication error** | Credentials invalid, session expired, or session absent.                                                                                                                                                                                  | Redirect to login screen. Show expiry message if session was previously valid.                                                                                 |
| **Authorization error**  | User is authenticated but lacks permission for the requested operation (e.g., changing another user's password).                                                                                                                          | Show error message. Do not redirect to login.                                                                                                                  |
| **Validation error**     | Request is malformed or violates business rules (e.g., transition from `erledigt`, missing required field, invalid dates).                                                                                                                | Show error message. Revert optimistic update if applicable.                                                                                                    |
| **Not found**            | The requested entity (project or user ID) does not exist.                                                                                                                                                                                 | Show error message. Re-fetch project list to sync state.                                                                                                       |
| **Route not found**      | The requested URL has no registered handler. Distinct from "Not found" above — the latter signals a known endpoint with an absent entity; this signals a stale client or a probe.                                                         | Show generic error message. Do not redirect.                                                                                                                   |
| **Gone**                 | The entity existed but is permanently unavailable; retry will not help (e.g., a Papierkorb row whose source bytes were lifecycle-reaped ahead of the row reaper, see [data-model.md §6.12](data-model.md#612-attachment-hidden-reaper)).  | Show error message. Do not retry.                                                                                                                              |
| **Conflict**             | The mutation cannot proceed because the resource state has changed since it was read (e.g., concurrent state transition) or the operation conflicts with related state (e.g., deleting a customer with active projects).                  | Refetch resource state. Optionally retry against the new state.                                                                                                |
| **Idempotency conflict** | A create request carried a client-supplied `id` that already identifies a row, but one or more participating fields in the request body differ from the stored row (see [§14.2.2](#1422-projects), [§14.2.5](#1425-customer-management)). | Close the create form, refresh the affected list, surface the message. Do not retry the same id with the same body — the request is ambiguous by construction. |
| **Server error**         | Unexpected internal failure.                                                                                                                                                                                                              | Show generic error message. Do not expose internal details.                                                                                                    |
| **Rate limited**         | Too many requests in the configured time window.                                                                                                                                                                                          | Show retry message. Back off before retrying.                                                                                                                  |

The full set of machine-readable error codes: `INVALID_CREDENTIALS`, `UNAUTHENTICATED`, `SESSION_EXPIRED`, `NOT_PERMITTED`, `VALIDATION_ERROR`, `NOT_FOUND`, `ROUTE_NOT_FOUND`, `GONE`, `CONFLICT`, `IDEMPOTENCY_CONFLICT`, `RATE_LIMITED`, `SCHEMA_VERSION_MISMATCH`, `TARGET_NOT_EMPTY`, `RESTORE_CONFIRMATION_MISMATCH`, `MISSING_USER_REFS`, `BULK_LIMIT_EXCEEDED`, `DEK_UNWRAP_FAILED`, `SERVER_ERROR`.

`MISSING_USER_REFS` is a specialization of the validation-error category — same client behavior (show error, no redirect, revert any optimistic UI), distinct code so the operator UI can render the targeted "Layer 2 restore required" remediation copy. See [§14.2.4](#1424-unified-data-exchange) for the validation semantics.

The error body uses the standard `{ code, message, details }` shape. The `details` payload for `MISSING_USER_REFS` is:

```typescript
{
  missingUserIds: string[];                            // deduplicated
  references: { path: string; userId: string }[];     // one entry per offending reference site
}
```

`missingUserIds` is the deduplicated list of user ids the operator must restore in the target before retrying. `references` carries one entry per offending envelope reference site (e.g., `{ path: "projects[3].createdBy", userId: "abc" }`) so the operator can locate every offending row without guessing; duplicate user-id references across different sites produce one entry per site. `MISSING_USER_REFS` is raised only when at least one referenced user id is absent in the target, so both fields are non-empty whenever the code is returned. The `path` shape (e.g. `customers[2].createdBy`, `project_workers[5].userId`) matches the intra-envelope issue path shape used for `VALIDATION_ERROR`. Error `details` keys in this spec are camelCase.

`DEK_UNWRAP_FAILED` is also a specialization of the validation-error category (`422`) — a per-row attachment envelope cannot be unwrapped against the currently-loaded binary identity (corrupt envelope or partial key rotation). The Service Worker translates the code to the [AC-244](verification.md#1526-attachments) "Schlüssel nicht verfügbar" placeholder render path. See [§14.2.11](#14211-attachments) for the unwrap semantics.

#### 14.4.2 Error Principles

- **Authentication errors are distinguishable from authorization errors.** An expired session must not produce the same error as an insufficient permission — the client handles them differently (redirect vs. inline message).
- **Validation errors are structured.** When a mutation is rejected for business rule violations, the error must identify which rule was violated (e.g., "cannot transition from terminal state") so the client can render a meaningful message.
- **Error messages never leak internal details.** No stack traces, database field names, table names, file paths, or query information in any error response. This applies to all error categories including server errors.
- **HTTP statusCode reflects the failure class.** A request rejected at the transport layer (malformed body declared as JSON, payload exceeding configured limits, unsupported media type, request to a URL with no registered handler) preserves the 4xx HTTP statusCode native to that failure and surfaces a stable machine-readable code. Transport-layer rejections are never collapsed into a 5xx response, and they are not logged at the operational `error` level (used to back 5xx alerting on logs). Rationale: 5xx must mean "the server failed", not "the client made a recoverable mistake"; conflating the two — either through the wire response or through the log severity — trains operators to ignore real 5xx incidents.
