# API Specification

---

## 14. API Specification

The API is the boundary between the front end and all persistent state ([architecture.md §11.1](architecture.md#111-mandatory-constraints)). The front end's state layer is a client of this API (see [architecture.md §11.3](architecture.md#113-state-layer-behavioral-contract)).

### 14.1 Design Principles

- **Single source of truth.** The API is authoritative for all persisted data.
- **Stack-agnostic contract.** Operations, inputs, and outputs — not transport, URLs, or serialization.
- **Consistent error representation.** Machine-readable code + human-readable message (see [§14.4](#144-error-handling)).
- **Stateless requests.** Each request carries all context needed, including authentication.
- **Pagination-ready.** List operations accept `offset`/`limit`. The contract must support pagination even if the current data volume does not require it.
- **Stable list order.** List operations return rows in a deterministic order — a second fetch yields the same sequence, and pagination pages do not overlap or skip rows.
- **Timestamps managed server-side.** `createdAt`, `updatedAt`, `statusChangedAt` are set by the server, never by clients.

Sets `updatedAt` and `updatedBy` server-side.                                                                |

**Project list filter parameters:** `status` (single or multiple), `customerId`, `plannedStartFrom`/`plannedStartTo` (date range), `hasNoDates` (boolean), `includeArchived` (boolean; default false; when true, includes `deleted = true` rows with archive state preserved), `search` (free-text across number, title, and customer name). Filters use AND logic. These filters serve both the project management view and the export operation.

Design notes:

- **Transitions are explicit operations**, not generic field updates — workflow adjacency rules are enforceable server-side.
- **Transition optimistic concurrency.** The client sends `expectedStatus` (the status it observed in its last read). The server advances only if the stored status still matches, otherwise the request is rejected with a `CONFLICT` error (see [§14.4](#144-error-handling)) — the client should refetch and present the current state. This deterministically prevents double-advance from sequential double-clicks, two tabs, or retried requests (see [data-model.md §6.4](data-model.md#64-concurrency) and [AC-94](verification.md#1517-data-integrity)).
- **Project number is immutable** after creation.
- **Soft delete**: no restore via the API. Deleted projects are excluded from all reads.
- **Purge is gated by archive-first.** A fresh project cannot be hard-deleted in a single step; the client must archive it (soft-delete via `DELETE /api/projects/:id`) before calling purge. This matches the trash-bin mental model and makes accidental destruction expensive (see [ADR-0017](../adr/0017-soft-delete-as-board-archive.md)).
- **Full project object returned** after every mutation so the client can reconcile without a separate fetch.
- **Update-dates** treats `plannedStart` and `plannedEnd` as a coordinated update. Clearing `plannedStart` also clears `plannedEnd` (see [data-model.md §6.8](data-model.md#68-date-validation)). `plannedEnd` before `plannedStart` is rejected.
- **Concurrent edit handling**: see [data-model.md §6.4](data-model.md#64-concurrency).
- The project response nests the full Customer object for rendering convenience.
- **Client-supplied `id` is optional.** When omitted, the server generates one. When supplied, it must be a syntactically valid id or the request is rejected as a validation error.
- **Idempotent replay.** If a row with the supplied id already exists AND every user-supplied field in the request body matches the stored row, the existing row is returned with the same success status as a fresh create. Replays do not create duplicates.
- **Field comparison for replay.** Participating fields: `number`, `title`, `customerId`, `status`, `plannedStart`, `plannedEnd`, `assignedWorkerIds`, `estimatedValue`, `notes`. Rules: `null`, `undefined`, and missing are equivalent; `assignedWorkerIds` compares as a set (order-independent); `estimatedValue` is rounded to two decimals before comparison; `plannedStart` and `plannedEnd` compare as calendar dates.
- **Idempotency conflict.** If a row with the supplied id exists but any participating field differs, the request is rejected with a conflict error code `IDEMPOTENCY_CONFLICT` (see [§14.4](#144-error-handling)). The stored row is not modified.
- **Soft-deleted match.** If the matching row is soft-deleted (see [data-model.md §6.9](data-model.md#69-soft-deletes)), the replay returns the row in its current (deleted) state. Clients must inspect the response rather than assume the row is active.
- **`number` uniqueness is separate.** Creating a project whose `number` collides with another row's `number` (regardless of idempotency) is rejected with a conflict error code `CONFLICT` and a German message naming the conflicting number (see [§14.4](#144-error-handling)). This is distinct from `IDEMPOTENCY_CONFLICT`.

#### 14.2.4 Unified Data Exchange

A single export and a single restore-only import cover the business-data layer (customers, projects, project-worker assignments). Per-entity bulk endpoints do not exist. Rationale and the three-layer persistence model live in [ADR-0018](../adr/0018-data-persistence-and-recovery-layered-strategy.md); the envelope shape is defined in [data-model.md §5.8](data-model.md#58-export-envelope).

| Operation  | Input                                                                                                                                             | Output                                                                        | Notes                                                                                                                                                                                                                                                                                                                                                                       |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Export** | —                                                                                                                                                 | unified envelope (see [data-model.md §5.8](data-model.md#58-export-envelope)) | Returns every customer, project, and project-worker assignment at row-level fidelity, including archived (soft-deleted) rows with archive state preserved. Users and sessions are excluded. Requires `data:export` permission.                                                                                                                                              |
| **Import** | unified envelope; optional `dry_run` flag; optional `override` flag; `confirmation_phrase` in the request body when overriding a non-empty target | `{ schema_version, summary }` on success; validation preview on dry-run       | Restore-only semantics. Applied as a single transaction — all-or-nothing. Rejects any `schema_version` not equal to the current one; no format migration is performed. Empty target proceeds; non-empty target is rejected unless `override` is set, which wipes existing business data atomically before restoring. IDs are preserved. Requires `data:restore` permission. |

Design notes:

- **Envelope shape** — `{ schema_version: int, exported_at: ISO 8601 string, customers: Customer[], projects: Project[], project_workers: { projectId, userId }[] }`. Each entity carries every persisted field including `id`, audit timestamps, and archive state (see [data-model.md §5.8](data-model.md#58-export-envelope)).
- **Dry-run mode** — the client signals dry-run via `?dry_run=true` on the import request. The server runs full validation against the envelope (schema version, intra-envelope referential integrity, missing-user references against the target database, row-level constraints), returns a preview describing what would be written, and performs no writes. A subsequent read reflects no state change. On the dry-run path both intra-envelope and missing-user-reference issues are evaluated regardless of intra-envelope state and surfaced together in the preview, so a single dry-run surfaces every blocker before commit.
- **Override flag** — the client signals override via `?override=true`. When the target is non-empty, override wipes existing customers, projects, and project-worker assignments atomically with the restore. Without the flag, a non-empty target rejects the request with a specific error code.
- **Confirmation phrase** — when `override=true` AND the target is non-empty, the request body MUST include `confirmation_phrase` equal to the configured phrase **[C]**. The server compares after trimming leading/trailing whitespace; comparison is case-sensitive. A missing or non-matching phrase is rejected with a specific error code and performs no writes. Dry-run requests and override-into-empty-target requests are exempt. The phrase is a destructive-action guard, not an authorization factor; the authoritative gate remains `data:restore`.
- **Missing user references** — the user-id fields carried in the envelope (`customers.createdBy`, `customers.updatedBy`, `projects.createdBy`, `projects.updatedBy`, `project_workers.userId`) are checked against the target database's `users` table during envelope validation. Null or missing values on the nullable audit fields (`createdBy`, `updatedBy` — see [data-model.md §5.1](data-model.md#51-project-entity), [§5.6](data-model.md#56-customer-entity), and the policy in [AC-98](verification.md#1517-data-integrity)) are not treated as references and do not trigger the check. A value is counted as a missing reference only when it is present in the envelope and absent from the target's `users` table. If any referenced user id is absent in the target, the import is rejected with error code `MISSING_USER_REFS` and performs no writes. The check runs on both the dry-run path (issues appear in the preview) and the commit path (request rejected), so an operator can detect the divergence under `dry_run=true` before committing to `override=true`. Rationale: the business-data layer is consistent with the layered persistence model (see [ADR-0018](../adr/0018-data-persistence-and-recovery-layered-strategy.md)). When the target's `users` table has diverged from the dump's user references (reseed, deleted worker, divergent installation), a clean fail tells the operator the right tool is the full DB restore (Layer 2), not the portable import. The error is not papered over by silently nulling the references.
- **Validation ordering** — on the commit path, intra-envelope referential integrity is reported first under `VALIDATION_ERROR`; the missing-user-reference check runs only when the envelope is intra-consistent, and `MISSING_USER_REFS` is raised only on an intra-consistent envelope — never both codes in one response. On the dry-run path this gate does not apply: both classes are evaluated and surfaced together in the preview regardless of intra-envelope state. Rationale: the two failures have distinct remediations (re-export the envelope vs. restore Layer 2 / reseed users), and on the commit path reporting them sequentially keeps each error response self-explanatory; the dry-run path remains the place to view both classes in parallel.
- **Atomic failure** — if any row fails validation during a non-dry-run import (with or without override), no state change is persisted. The request has no partial outcome. The missing-user-reference check participates in this guarantee: detection happens in envelope validation, before any wipe or insert runs.
- **Schema version** — a monotonic integer. Exports stamp the current value; imports compare and reject mismatches outright (see [ADR-0018 §Decision](../adr/0018-data-persistence-and-recovery-layered-strategy.md#decision)).
- **Users and sessions are not exchanged** through this surface. Admin bootstrap ([ADR-0010](../adr/0010-first-run-admin-bootstrap.md)) creates the first user; test seeding uses a direct-DB helper confined to the test layer.

#### 14.2.6 Data Extraction

All extraction operations require an authenticated session and the `customer:write` permission. See [ADR-0016](../adr/0016-llm-email-extraction-via-server-proxied-openrouter.md) for the server-side proxy rationale.

| Operation         | Input                  | Output                                                                                                                   | Notes                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ----------------- | ---------------------- | ------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Extract email** | `text`: raw email text | `{ customer: { name, phone, email, street, zip, city }, project: { title, description } }` — all fields nullable strings | Routes the text to an external LLM via a server-side proxy. The text must be non-empty and not exceed 50,000 characters. Design notes:

- **Stateless with respect to persistence.** The extraction endpoint does not create, update, or delete any record. It is a pure read: email text in, structured data out.

- **Every field may be null.** The LLM returns `null` for information it could not find in the input. Clients must handle nulls across all fields.

#### 14.2.7 Backup Status

The Layer 2 backup-freshness badge ([verification.md §15.22](verification.md#1522-backup-and-recovery), [ADR-0020](../adr/0020-layer-2-encrypted-r2-backups-with-operator-loaded-drills.md)) needs a read surface on the unauthenticated login screen. This endpoint publishes the [`BackupStatus`](data-model.md#59-backup-status-entity) fields and nothing else; the network reach of the app is VPN-gated ([ADR-0008](../adr/0008-vpn-first-network-access.md)), which is the threat-model anchor for exposing it publicly.

| Operation             | Input | Output                                                                                                                                      | Notes                                                                                                                                                                                             |
| --------------------- | ----- | ------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Get backup status** | —     | `{ available: true, status: BackupStatus }` on success; `{ available: false }` when the row is unreachable (DB down and mirror unavailable) | No authentication required. Returns only the fields defined in [data-model.md §5.9](data-model.md#59-backup-status-entity). Rate-limited to protect the DB path on a flood of login-screen loads. |

Design notes:

- **Response contract is an explicit allowlist.** The server returns only the badge-visible fields from [data-model.md §5.9](data-model.md#59-backup-status-entity) (`lastBackupAt`, `lastBackupOk`, `lastDrillAt`, `lastDrillOk`, `lastError`, `updatedAt`). Any future sensitive column on the `meta_backup_status` row does not leak through this surface unless it is explicitly added.
- **Unreachable source → `{ available: false }`, not silence.** Misleading state is a critical defect class ([ADR-0014](../adr/0014-ac-tier-system-critical-vs-design.md)). An unreachable DB yields an explicit response the client renders as "Status unbekannt" ([AC-171](verification.md#1522-backup-and-recovery)) rather than a 404 or a silent empty payload.
- **No errors on absence of prior runs.** The pre-seed row from the baseline migration is always present. The service reports `lastBackupOk=false` with `lastBackupAt=null` for a DB that has never run a backup; the client derives the `backup-never-run` badge reason from that shape.

#### 14.2.8 Audit Log

A read-only surface over [`audit_log`](data-model.md#510-audit-log-entity). The table is append-only ([data-model.md §6.10](data-model.md#610-audit-log-retention)); no create, update, or delete operation is exposed.

| Operation              | Input                                                                                                                                           | Output                                 | Notes                                                                                                                                                                                                                                                             |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **List audit entries** | optional: offset, limit, filters: `entityType`, `entityId`, `ancestorType`, `ancestorId`, `entityLabelQuery`, `actorId`, `from`, `to`, `action` | audit entry list, total count          | Returns entries visible to the authenticated caller per the scoping rules below. Ordered by `createdAt` descending (newest first), stable tiebreaker `id`. Default page size matches other list endpoints. Requires `audit:read` permission. Filters AND-compose. |
| **Get audit entry**    | audit entry id                                                                                                                                  | single entry, or not-found / forbidden | Returns the full entry subject to the scoping rules below. Requires `audit:read` permission.                                                                                                                                                                      |

Design notes:

- **RBAC-sliced reads.** A single scope predicate — `auditDestructiveScopeForCaller(user)` — composes at the repository layer, following the pattern established in [ADR-0019](../adr/0019-worker-data-scoping-repository-layer-predicate.md) (permissions stay coarse; scope is orthogonal to capability). For owner the predicate returns null (no filter); for every other role that holds `audit:read` (office under the current matrix) it returns a `WHERE` fragment excluding destructive entries, defined as: `action = 'purge'` on any entity type; `action = 'delete'` AND `entityType = 'user'`; `action = 'update'` AND `entityType = 'user'` where the payload diff touches `roles`. The predicate AND-composes with the permission gate. An earlier revision also carried a reachability predicate for workers (self-authored + project-reachability); it was removed when workers lost `audit:read` — the audit surface is administrative, not worker-facing.
- **Three-way result (same as ADR-0019).** Get-by-id distinguishes three outcomes: not-found (`404 NOT_FOUND`), in-scope (`200`), out-of-scope (`403 NOT_PERMITTED` — only the destructive-action narrowing can produce this, and only for office). Out-of-scope must not be collapsed to not-found — the caller-identity boundary matters for the same reasons as in [AC-147](verification.md#1521-role-scoping).
- **Actor name in response.** For a `user`-actor entry, the response includes the acting user's `displayName` as a derived read-time convenience — independent of whether the caller holds `user:read`. The field is a presentation aid for the activity feed; it is not a substitute for the `user` list endpoint and carries no additional user-record fields. System-actor entries include `actorReason` instead.
- **Entity filters.** `entityId` takes a UUID and matches rows equal to that id. `ancestorType` + `ancestorId` take the same shape and match rows whose write-time ancestor pair equals `(ancestorType, ancestorId)` ([architecture.md §11.12](architecture.md#1112-audit-ancestor-link)) — this is the filter the project-detail activity feed uses so the one call returns project rows plus every nested-entity row (`project_worker`, `attachment`) scoped to that project. `entityLabelQuery` takes a case-insensitive substring of at least three characters and matches rows whose frozen `entityLabel` contains it — it powers the Aktivität filter bar, where the caller types a name. LIKE metacharacters are escaped (`%`, `_`, `\`) so user input is treated literally. Rows whose `entityLabel` is null (import and retention paths that cannot supply a label) are excluded from `entityLabelQuery` matches by construction. The substring match is backed by a pg_trgm GIN index on `entity_label`; sub-three-character queries are rejected (422) to keep the index useful. Ancestor filters are served by the compound index `audit_log_ancestor_idx`.
- **Payload drawer.** Entries include `payload` as returned. The client renders the diff drawer only on rows with a non-null, non-empty payload.
- **Pagination and ordering.** Entries are ordered newest-first by `createdAt`, with `id` as the deterministic tiebreaker. A second fetch with the same filters yields the same sequence; pages do not overlap or skip rows (per [§14.1](#141-design-principles)).
- **Error paths** (mapped to the categories in [§14.4.1](#1441-error-categories)):
  - Missing or expired session → `401 UNAUTHENTICATED` / `SESSION_EXPIRED` (authentication error).
  - Authenticated caller lacks `audit:read` → `403 NOT_PERMITTED` (authorization error).
  - Get-by-id on an existing row outside the caller's scope → `403 NOT_PERMITTED` (authorization error — see three-way result above).
  - Get-by-id on a non-existent row → `404 NOT_FOUND`.
  - Filter or pagination parameters failing schema validation → `422 VALIDATION_ERROR`.
  - `from`/`to` inverted (`to < from`) → `422 VALIDATION_ERROR`.
  - `entityType`, `ancestorType`, or `action` filter values outside the documented enum / vocabulary ([data-model.md §5.10](data-model.md#510-audit-log-entity)) → `422 VALIDATION_ERROR`.
  - Request using an HTTP method other than the documented reads on the audit endpoints → `405 METHOD_NOT_ALLOWED`.
- **Post-commit publisher contract.** The notification publisher module exposes:
  - `onAuditCommitted(handler)` — registers a handler invoked with the committed audit entry after the domain transaction commits. Handlers are invoked once per commit, in registration order.
  - `setOperationalLogger(logger)` — wires the structured operational logger used to surface handler failures.
    A throwing handler does not roll back the originating mutation — see [AC-183](verification.md#1523-audit-log) for the failure-surface contract. The publisher is process-local; restart-durability is out of scope (in-process dispatch per [ADR-0021](../adr/0021-audit-log-and-notifications-single-write-path.md)).

#### 14.2.9 Notification Rules

Admin-editable rules mapping the closed event catalog ([data-model.md §5.11](data-model.md#511-notification-rule)) to recipient specs. CRUD is gated on `notifications:manage` and follows the validation rules below ([ADR-0023](../adr/0023-notification-rules-db-stored-closed-event-catalog.md)).

| Operation       | Input                                                        | Output                 | Notes                                                                                                                                                                                  |
| --------------- | ------------------------------------------------------------ | ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **List rules**  | optional: offset, limit                                      | rule list, total count | All rules. Ordered by `eventClass` ascending, `createdAt` ascending as tiebreaker. Requires `notifications:manage`.                                                                    |
| **Get rule**    | rule id                                                      | single rule            | Full rule or not-found. Requires `notifications:manage`.                                                                                                                               |
| **Create rule** | eventClass, stateFilter?, recipientSpec, enabled             | created rule           | `recipientSpec` must carry at least one non-empty part. `stateFilter` permitted only on `project.transition_forward` / `project.transition_backward`. Requires `notifications:manage`. |
| **Update rule** | rule id, eventClass?, stateFilter?, recipientSpec?, enabled? | updated rule           | PATCH — omitted fields unchanged. Same validation as create. `eventClass` is updatable. Requires `notifications:manage`.                                                               |
| **Delete rule** | rule id                                                      | success/failure        | Hard-deletes. No soft-delete — rules are configuration. Requires `notifications:manage`.                                                                                               |

Design notes:

- **Validation rules** (all raise validation error on breach):
  - `eventClass` must be a member of [`NotificationEventClass`](data-model.md#511-notification-rule).
  - `stateFilter` must be null unless `eventClass` is a transition event; when present, it must be a member of the configured workflow-state set ([index.md §3](index.md#3-workflow-states)).
  - `recipientSpec.roles` must be a subset of the configured role set ([index.md §4.2](index.md#42-users)); `userIds` must each reference an active `UserAccount`; `includeAssignedWorkers` is rejected when `eventClass` is a non-project-scoped event (`backup.failed`, `disk.threshold_reached`).
  - At least one of `roles`, `includeAssignedWorkers`, or `userIds` must resolve to a non-empty part — an enabled rule with an empty spec is rejected.
- **Error paths** (mapped to the categories in [§14.4.1](#1441-error-categories)):
  - Missing or expired session → `401 UNAUTHENTICATED` / `SESSION_EXPIRED`.
  - Caller lacks `notifications:manage` → `403 NOT_PERMITTED`.
  - Get / update / delete on an unknown id → `404 NOT_FOUND`.
  - Validation breaches above → `422 VALIDATION_ERROR`.
- **Take-effect semantics.** A successful create, update, or delete affects the next event committed after the change; in-flight events use the rule set read at their own commit.
- **No uniqueness constraint on `(eventClass, stateFilter, recipientSpec)`.** Duplicates are permitted; the publisher's recipient-union dedups by `user_id` at resolution. A uniqueness constraint would force admins to reason about rule-equivalence — exactly the complexity the flat model avoids ([ADR-0023](../adr/0023-notification-rules-db-stored-closed-event-catalog.md)).

#### 14.2.10 Push Subscription

Browser push subscription management. Authenticated self-scope — a caller manages only their own subscriptions. The VAPID public-key read is the only unauthenticated route in this section (the key is intentionally public; the private half never leaves the server).

| Operation                 | Input                                     | Output                             | Notes                                                                                                                                                         |
| ------------------------- | ----------------------------------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Read VAPID public key** | —                                         | `{ vapidPublicKey: string\|null }` | Unauthenticated. Returns the server-derived VAPID public key, or `null` when VAPID credentials are not configured on the server. GET-only; non-GET verbs 405. |
| **Subscribe to push**     | endpoint, keys (p256dh, auth), userAgent? | success/failure                    | Registers a subscription for the authenticated caller. Re-subscribing an existing `endpoint` updates the row rather than duplicating.                         |
| **Unsubscribe from push** | endpoint (or subscription id)             | success/failure                    | Removes one of the caller's subscriptions. Unknown endpoints / ids are idempotent no-ops.                                                                     |

Design notes:

- **VAPID key is server-sourced.** The client fetches the public key at subscribe time from the read endpoint; the server derives it from the private half at startup, so the operator maintains a single server-side secret. Short HTTP cache (5 minutes) — a deploy rotates the key and the browser cache expires well inside one user session.
- **Ownership is derived.** The server sets `userId` from the session; a client-supplied user id is ignored.
- **Self-scoped lookups.** Subscribe / unsubscribe filter by `(subscription.id OR endpoint, userId = session.userId)` before acting. An id or endpoint belonging to another user is treated as unknown — the no-op response is indistinguishable from a truly-unknown one. Rationale: self-scope endpoints must not enumerate other users' subscriptions.
- **No auto-request.** The permission prompt is user-initiated from the settings affordance ([ui/index.md §8.7.2](ui/index.md#872-user-menu)); the server never pushes a registration on login.
- **Dispatch-time pruning.** A subscription the browser reports permanently dead at dispatch is removed server-side — housekeeping, not a user-facing operation.
- **Mute interaction.** `pushMuted = true` suppresses delivery to every subscription the user owns; rows are retained, so unmuting restores delivery without a re-subscribe.
- **Rate limiting.** Mutation paths (`POST`, `DELETE`) share a per-session mutation bucket ([architecture.md §12.2 rate-limit config](architecture.md#122-company-configurable-settings) `[C]`); exceeding returns `429 RATE_LIMITED`. The VAPID public-key read is not rate-limited (unauthenticated, GET-only, 5-minute cache).
- **Error paths** (mapped to the categories in [§14.4.1](#1441-error-categories)):
  - Missing or expired session → `401 UNAUTHENTICATED` / `SESSION_EXPIRED` (subscribe / unsubscribe only; the VAPID read is unauthenticated).
  - Malformed `endpoint` / `keys` → `422 VALIDATION_ERROR`.
  - `endpoint` exceeds 2048 characters, `keys.p256dh` or `keys.auth` exceeds 512 characters, or `endpoint` is not a syntactically valid URI → `422 VALIDATION_ERROR` (bounded to protect the subscription table from pathological input).

#### 14.2.11 Attachments

Per-project file attachments. The app never handles attachment bytes — uploads and downloads go directly between the client and object storage via server-issued presigned URLs, each constrained by a signed policy that pins the exact object key, content type, size ceiling, and expiry. Metadata is the `Attachment` entity ([data-model.md §5.13](data-model.md#513-attachment)).

| Operation                 | Input                                                                           | Output                                                                                                                                           | Notes                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ------------------------- | ------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **List attachments**      | project ID                                                                      | list of `Attachment` rows (see [data-model.md §5.13](data-model.md#513-attachment)); `pending` rows excluded                                     | Returns only `status = 'ready'` rows for the project. Ordered by `createdAt` descending, `id` as tiebreaker. Requires `attachment:read` and a caller in scope for the project (see scoping below).                                                                                                                                                                                                                                                                   |
| **Init upload**           | project ID, `fileName`, `mimeType`, `sizeBytes`, `label`, `hasThumbnail` (bool) | `{ attachment: Attachment, originalUpload: PresignedPost, thumbnailUpload?: PresignedPost }`                                                     | Creates a `pending` attachment row and returns two presigned-POST descriptors (or one when the MIME type is a non-photo kind). The server fixes the object key, content type, and size ceiling in the policy conditions — the client cannot alter them. Requires `attachment:write` and the project to be in scope. Presigned URLs expire per the configured presigned-URL TTL **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings)). |
| **Complete upload**       | attachment ID                                                                   | updated `Attachment` with `status = 'ready'`                                                                                                     | Verifies via a HEAD against object storage that both the original and (for photos) the thumbnail exist, that the reported content type matches, and that the reported size does not exceed the cap **[C]**. On success flips `status` to `'ready'`. On HEAD failure the row stays `pending` and the request returns a conflict error.                                                                                                                                |
| **Delete attachment**     | attachment ID                                                                   | success/failure                                                                                                                                  | Hard-deletes the row and deletes both backing objects from storage (single logical operation; partial failure reverts per error paths below). Permission gate per the delete matrix below.                                                                                                                                                                                                                                                                           |
| **Get download URL**      | attachment ID, `variant` (`'original'` \| `'thumbnail'`)                        | `{ url: string, expiresAt: string }`                                                                                                             | Returns a presigned-GET URL for the requested variant whose lifetime is the configured presigned-URL TTL **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings)). Requires `attachment:read` and caller in project scope. Thumbnail requests on a non-photo attachment return a validation error.                                                                                                                                       |
| **Get bulk download URL** | project ID, `attachmentIds: string[]`                                           | `{ url: string, expiresAt: string }` on success; `{ code: 'BULK_LIMIT_EXCEEDED', limits: { maxFiles, maxBytes } }` on cap breach (details field) | Streams a zip of the requested attachments via a single presigned URL. Capped at the configured bulk-download limits **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings)) — exceeding either cap is rejected as a validation error. Requires `attachment:read` and caller in project scope. All requested IDs must belong to the supplied project.                                                                                   |

```typescript
interface PresignedPost {
  url: string; // POST target
  fields: Record<string, string>; // form fields the client must echo verbatim
  expiresAt: string; // ISO 8601 — after this the descriptor is useless
}
```

Design notes:

- **No byte traffic through the app.** Upload bytes travel client → object storage; download bytes travel object storage → client. The application sees only the metadata row and the HEAD check at complete time. This is what lets the app stay small regardless of attachment volume.
- **Presigned-POST policy conditions** — every init issues a policy that pins: exact `originalKey` / `thumbKey` (no wildcards), a `content-type starts-with` matching the requested MIME, a `content-length-range` with upper bound equal to the size cap **[C]**, and an expiry equal to the configured presigned-URL TTL **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings)). A client that mutates any conditioned field causes the storage provider to reject the upload with a signature error; the app never observes a mismatched object.
- **Complete is retry-safe while the row is at `pending`.** A 409 from a HEAD miss leaves the row retry-eligible; a subsequent complete with both objects now HEAD-present succeeds. The one-way `pending → ready` transition (see [AC-212](verification.md#1526-attachments)) is what makes this safe: the row cannot be finalized twice, and a second complete on a `ready` row is itself rejected with 409.
- **Client-generated thumbnail.** For photo MIME types the client produces a thumbnail before calling init and sends `hasThumbnail = true`. The server issues two presigned descriptors; the client posts both. The server does not transform bytes. The thumbnail format and the sizing parameters live in `ARCHITECTURE.md § Attachments Module` (impl-specific).
- **Labels and MIME whitelist.** Init validates `label` against the closed enum ([data-model.md §5.13](data-model.md#513-attachment)) and `mimeType` against the whitelist. Outside values produce a validation error with no row persisted.
- **Scoping — repository-layer predicate, parallel to [ADR-0019](../adr/0019-worker-data-scoping-repository-layer-predicate.md).** `attachmentScopeForCaller(user)` returns `null` for owner, office, bookkeeper; for worker it returns a `WHERE` fragment admitting only attachments whose project the worker appears in (`project_workers`). The predicate is AND-composed with the permission gate. Get-by-id, like the project get-by-id pattern, distinguishes three outcomes: `404 NOT_FOUND`, `200` in-scope, `403 NOT_PERMITTED` out-of-scope (the existence-through-role-boundary policy in [AC-147](verification.md#1521-role-scoping) applies identically).
- **Worker write rules.** A worker may call init + complete only on a project where the worker is in `project_workers`; otherwise the request is rejected with `403 NOT_PERMITTED`. A worker may delete only attachments where `createdBy = callerId` AND `now - createdAt` is within the configured self-delete grace window **[C]** ([architecture.md §12.2](architecture.md#122-company-configurable-settings)). Outside this window, or for attachments the worker did not author, the delete is rejected with `403 NOT_PERMITTED`. Owner and office hold `attachment:delete` without the author or time constraint.
- **Bookkeeper read-only.** Bookkeeper holds `attachment:read` (unscoped, same MVP stand-in as `project:read` and `customer:read`) but no write or delete.
- **Mutation auditing.** Init (create) and delete go through the single-write-path helper and produce `audit_log` rows ([data-model.md §5.10](data-model.md#510-audit-log-entity), [AC-177](verification.md#1523-audit-log)). The action vocabulary adds `attachment:add` and `attachment:remove` (init and delete respectively). Complete is a state-machine finalize on the same row init created; it does not produce an additional audit row — the `attachment:add` entry is the authoritative record of the attachment entering the project. `entityType` is `'attachment'` and `entityId` is the attachment's `id`; the write-time ancestor is `('project', projectId)` ([architecture.md §11.12](architecture.md#1112-audit-ancestor-link)) so the per-project activity feed's `ancestorType + ancestorId` filter picks the row up alongside project- and `project_worker`-scoped rows in one indexed query. `payload` retains `projectId` for backward-compatible drawer rendering.
- **Bulk-download filename collisions.** Zip entries carry each attachment's `fileName`; on collisions the server deduplicates (implementation-defined) so the archive extracts without overwrites. The disambiguation strategy is not part of the contract — clients must not rely on a specific scheme.
- **Pending rows are ineligible for bulk download.** The 422 on a `pending` id (see Error paths below, and [AC-216](verification.md#1526-attachments)) is a correctness guard, not an authorization check: a `pending` row's backing objects may not yet be durable (HEAD may still fail), so including one in a zip risks a broken entry. On rejection the client must refetch the list and exclude any row still at `pending` before retrying.
- **Error paths** (mapped to the categories in [§14.4.1](#1441-error-categories)):
  - Missing or expired session → `401 UNAUTHENTICATED` / `SESSION_EXPIRED`.
  - Caller lacks the relevant permission → `403 NOT_PERMITTED`.
  - Caller out of scope for the project (worker path) → `403 NOT_PERMITTED` on a row that exists; `404 NOT_FOUND` on a missing id — distinguishable, see [AC-147](verification.md#1521-role-scoping).
  - Init: `mimeType` outside whitelist, `label` outside enum, `sizeBytes` exceeds the configured per-file cap **[C]**, or `fileName` violating the sanitization rules (empty, > 255 characters, contains a control character `\x00`–`\x1F`/`\x7F`, path separator `/`/`\`, or double-quote `"`) → `422 VALIDATION_ERROR`. No row persisted. The double-quote ban is load-bearing: `fileName` is interpolated into the `Content-Disposition` header of presigned-GET download responses (see [data-model.md §5.13](data-model.md#513-attachment) "Filename sanitization").
  - Complete: HEAD finds an object missing, size mismatches the row's `sizeBytes`, or content type mismatches `mimeType` → `409 CONFLICT`. The row stays `pending`; the reaper ([data-model.md §6.11](data-model.md#611-attachment-orphan-reaper)) is the only path that removes it.
  - Complete on a row already at `status = 'ready'` → `409 CONFLICT`. No state change, no duplicate HEAD verification — the transition is one-way and non-replayable.
  - Complete on an id the reaper has already removed (row gone) → `404 NOT_FOUND`. The client discards the pending upload state and does not retry.
  - Presigned-POST expiry mid-upload: if the upload does not complete within the presigned-URL TTL **[C]**, the storage provider rejects the POST with a signature/time error. The app never observes the expired upload. The `pending` row stays in place; the orphan reaper ([data-model.md §6.11](data-model.md#611-attachment-orphan-reaper)) removes it once the row's age passes the reaper TTL. Retry is a fresh init call (new row, new presigned descriptors); reusing the original `attachmentId` after presigned expiration is not supported.
  - Delete: row already deleted → `404 NOT_FOUND`. Worker outside the self-delete grace → `403 NOT_PERMITTED`. Storage delete failure (object missing) → success; storage delete failure (transient) → `500 SERVER_ERROR` and the row is not removed (atomicity with storage).
  - Bulk download: requested count or summed `sizeBytes` exceeds the configured bulk-download caps **[C]** → `422 VALIDATION_ERROR` with `code = BULK_LIMIT_EXCEEDED` and a `limits` field naming the caps. Any id not belonging to the supplied project, OR any id referencing a row with `status = 'pending'` → `422 VALIDATION_ERROR`. The whole batch is rejected; no partial-serve.
  - Download-URL: unknown variant → `422 VALIDATION_ERROR`. Thumbnail on a non-photo row → `422 VALIDATION_ERROR`.
  - A request using an HTTP method other than the documented verbs on any attachment endpoint → `405 METHOD_NOT_ALLOWED`.

### 14.3 Authorization Rules

- All API operations require authentication (valid, active session).
- The system implements a basic role-based permission matrix. All authenticated, active users can view all projects (list, get) and change their own password. Other operations — including mutations, imports, and exports — require specific permissions granted by role:

| Role       | Permissions                                                                                                                                                                                                                                                                                                                                  |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| owner      | project:read, project:transition, project:dates, project:create, project:update, project:delete, project:purge, customer:read, customer:write, customer:delete, user:read, user:manage, user:delete, data:export, data:restore, audit:read, notifications:manage, attachment:read, attachment:write, attachment:delete, auth:change-password |
| office     | project:read, project:transition, project:dates, project:create, project:update, project:delete, customer:read, customer:write, user:read, data:export, audit:read, attachment:read, attachment:write, attachment:delete, auth:change-password                                                                                               |
| worker     | project:read, customer:read, attachment:read, attachment:write, attachment:delete, auth:change-password                                                                                                                                                                                                                                      |
| bookkeeper | project:read, customer:read, attachment:read, auth:change-password                                                                                                                                                                                                                                                                           |

Design notes:

- **Data exchange permissions are distinct from entity read/write permissions.** `data:export` gates the unified export ([§14.2.4](#1424-unified-data-exchange)); `data:restore` gates the unified import. Granting `project:read` does not imply `data:export`.
- **`notifications:manage` is admin-only.** The rule-editor surface ([§14.2.9](#1429-notification-rules)) is admin-only (owner under the default matrix, per [ADR-0023](../adr/0023-notification-rules-db-stored-closed-event-catalog.md)); rule changes affect who-sees-what platform-wide, so this aligns with `user:manage`.
- **`data:restore` is owner-only.** The import destroys or replaces business data in a single transaction — a destructive operation, reserved to the most privileged role consistent with `customer:delete`, `user:delete`, and `project:purge`.
- **Audit read is a single permission; visibility is narrowed by orthogonal scope predicates at the repository layer.** `audit:read` admits a caller to the audit surface at all. Destructive-action visibility is not a second permission: it is the `auditDestructiveScopeForCaller` predicate defined in [§14.2.8](#1428-audit-log) — owner gets a null filter; every other role gets a `WHERE` fragment excluding destructive entries. This follows [ADR-0019](../adr/0019-worker-data-scoping-repository-layer-predicate.md)'s decision that permissions stay coarse and scope is orthogonal to capability — encoding scope in the permission catalog would conflate _capability_ with _extent_.
- The API must be designed so that scoping reads to a subset (e.g. "projects assigned to user X") is an additive query filter on the list operation, not a restructuring of the endpoint.
- **Holding `project:read` does not imply unscoped access to every project; holding `customer:read` does not imply unscoped access to every customer.** For workers, reads (list and get) are scoped by caller identity — restricted to projects where the worker is in `project_workers`, and to customers referenced by those projects. Owner and office remain unscoped. Exactly one `project:read` and one `customer:read` exist in the matrix — this is observable in the table above. Bookkeeper has unscoped project and customer read as an MVP placeholder until an invoice-oriented view is introduced (see [index.md §4.2](index.md#42-users)). See also [AC-145](verification.md#1521-role-scoping), [AC-146](verification.md#1521-role-scoping), [AC-147](verification.md#1521-role-scoping), [AC-148](verification.md#1521-role-scoping).
- **Attachment permissions mirror the scope-orthogonal pattern.** `attachment:read`, `attachment:write`, and `attachment:delete` are coarse capabilities; visibility is narrowed by `attachmentScopeForCaller` at the repository layer ([§14.2.11](#14211-attachments)), following [ADR-0019](../adr/0019-worker-data-scoping-repository-layer-predicate.md). Worker write is additionally gated at the service layer on the project being in the worker's scope; worker delete is additionally gated on authorship plus a self-delete grace ([§14.2.11](#14211-attachments)). Owner and office remain unscoped, consistent with their project and customer reach.
- Authorization checks must be enforced server-side. Client-side UI hiding (e.g. hidden buttons) is a UX convenience, not a security measure (see [architecture.md §13.6](architecture.md#136-security)).

### 14.4 Error Handling

Every error response carries two components:

| Component                  | Purpose                               | Example                                                   |
| -------------------------- | ------------------------------------- | --------------------------------------------------------- |
| **Machine-readable code**  | Programmatic handling by the client   | `INVALID_CREDENTIALS`, `SESSION_EXPIRED`, `NOT_PERMITTED` |
| **Human-readable message** | Display to the user (German, **[C]**) | `"Anmeldung fehlgeschlagen"`, `"Sitzung abgelaufen"`      |

An optional third component, `details`, may carry structured validation information (e.g., field-level errors from schema validation). Clients should handle its absence gracefully.

#### 14.4.1 Error Categories

The API must distinguish the following error categories. Each category has distinct client-side handling (see [ui/behavior.md — Asynchronous Mutation Behavior](ui/behavior.md#95-asynchronous-mutation-behavior)).

| Category                 | Meaning                                                                                                                                                                                                                                   | Client behavior                                                                                                                                                |
| ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Authentication error** | Credentials invalid, session expired, or session absent.                                                                                                                                                                                  | Redirect to login screen. Show expiry message if session was previously valid.                                                                                 |
| **Authorization error**  | User is authenticated but lacks permission for the requested operation (e.g., changing another user's password).                                                                                                                          | Show error message. Do not redirect to login.                                                                                                                  |
| **Validation error**     | Request is malformed or violates business rules (e.g., transition from `erledigt`, missing required field, invalid dates).                                                                                                                | Show error message. Revert optimistic update if applicable.                                                                                                    |
| **Not found**            | The requested entity (project or user ID) does not exist.                                                                                                                                                                                 | Show error message. Re-fetch project list to sync state.                                                                                                       |
| **Conflict**             | The mutation cannot proceed because the resource state has changed since it was read (e.g., concurrent state transition) or the operation conflicts with related state (e.g., deleting a customer with active projects).                  | Refetch resource state. Optionally retry against the new state.                                                                                                |
| **Idempotency conflict** | A create request carried a client-supplied `id` that already identifies a row, but one or more participating fields in the request body differ from the stored row (see [§14.2.2](#1422-projects), [§14.2.5](#1425-customer-management)). | Close the create form, refresh the affected list, surface the message. Do not retry the same id with the same body — the request is ambiguous by construction. |
| **Server error**         | Unexpected internal failure.                                                                                                                                                                                                              | Show generic error message. Do not expose internal details.                                                                                                    |
| **Rate limited**         | Too many requests in the configured time window.                                                                                                                                                                                          | Show retry message. Back off before retrying.                                                                                                                  |

The full set of machine-readable error codes: `INVALID_CREDENTIALS`, `UNAUTHENTICATED`, `SESSION_EXPIRED`, `NOT_PERMITTED`, `VALIDATION_ERROR`, `NOT_FOUND`, `CONFLICT`, `IDEMPOTENCY_CONFLICT`, `RATE_LIMITED`, `SCHEMA_VERSION_MISMATCH`, `TARGET_NOT_EMPTY`, `RESTORE_CONFIRMATION_MISMATCH`, `MISSING_USER_REFS`, `BULK_LIMIT_EXCEEDED`, `SERVER_ERROR`.

`MISSING_USER_REFS` is a specialization of the validation-error category — same client behavior (show error, no redirect, revert any optimistic UI), distinct code so the operator UI can render the targeted "Layer 2 restore required" remediation copy. See [§14.2.4](#1424-unified-data-exchange) for the validation semantics.

The error body uses the standard `{ code, message, details }` shape. The `details` payload for this code is:

```typescript
{
  missingUserIds: string[];                            // deduplicated
  references: { path: string; userId: string }[];     // one entry per offending reference site
}
```

`missingUserIds` is the deduplicated list of user ids the operator must restore in the target before retrying. `references` carries one entry per offending envelope reference site (e.g., `{ path: "projects[3].createdBy", userId: "abc" }`) so the operator can locate every offending row without guessing; duplicate user-id references across different sites produce one entry per site. `MISSING_USER_REFS` is raised only when at least one referenced user id is absent in the target, so both fields are non-empty whenever the code is returned. The `path` shape (e.g. `customers[2].createdBy`, `project_workers[5].userId`) matches the intra-envelope issue path shape used for `VALIDATION_ERROR`. Error `details` keys in this spec are camelCase.

#### 14.4.2 Error Principles

- **Authentication errors are distinguishable from authorization errors.** An expired session must not produce the same error as an insufficient permission — the client handles them differently (redirect vs. inline message).
- **Validation errors are structured.** When a mutation is rejected for business rule violations, the error must identify which rule was violated (e.g., "cannot transition from terminal state") so the client can render a meaningful message.
- **Error messages never leak internal details.** No stack traces, database field names, table names, file paths, or query information in any error response. This applies to all error categories including server errors.
